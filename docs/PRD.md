# PRD: geowiki

版本：v1.0
日期：2026-07-12
状态：草案

## 1. 产品概述

geowiki 是一款专注地质领域的百科全搜，面向矿产资源、地质勘查、矿山设计和自然资源管理场景提供专业知识问答。

用户通过自然语言提出专业问题，系统基于本地知识库、全文搜索、向量检索、知识图谱和有限联网补充能力，返回有依据、可追溯、可核验的答案。

第一阶段产品目标不是做通用聊天机器人，而是解决“专业问题应该依据哪个标准、哪一条、适用于什么条件”的检索和解释问题。

## 2. 背景与问题

矿产资源领域的标准、规范、规程、政策文件数量多，更新周期长，分布来源复杂。实际工作中，技术人员经常需要回答类似问题：

- 某个矿种资源量估算应依据哪个规范？
- 某个术语在标准中如何定义？
- 某类工程间距、勘查类型、控制程度要求在哪个条款中规定？
- 某个标准是否现行，是否存在替代或修改单？

当前主要痛点：

- 标准文件分散，人工检索耗时。
- 关键词检索命中多，但难以快速定位条款。
- 大模型容易在没有证据时生成看似合理但不可核验的答案。
- 官方网站常常只提供元数据，正文可能是图片式预览，文本模型无法直接读取。
- 行标、国标、地方标准和第三方文档之间存在版本、状态和来源可信度差异。

## 3. 产品目标

MVP 阶段目标：

- 提供一个 API-first 的专业问答能力，前端主要用于内测、演示和管理。
- 对每个答案展示引用来源、条款或章节、页码、原文片段或证据摘要。
- 在证据不足时明确说明不足，不编造标准条款。
- 支持本地知识库优先，联网搜索作为补充。
- 为后续接入标准库、OCR、知识图谱、配额统计和开发者 API 打好接口基础。
- 支持开源本地版、云端受控测试版和机构内部部署版三种演进方向。

非目标：

- 不做通用办公聊天助手。
- 不回答与矿产资源、地质勘查、矿山设计、自然资源管理、标准规范或相关政策无关的问题。
- 不替代正式标准出版物和专业人员审查。
- 不在 MVP 阶段做完整知识库后台管理。
- 不在 MVP 阶段做多租户、付费、复杂审批流。
- 不在开源仓库中内置真实标准全文、OCR 后全文或预构建真实标准知识库。

## 4. 目标用户

### 4.1 一线技术人员

包括地勘单位、矿山企业、设计院和咨询单位技术人员。

核心需求：

- 快速找到依据标准。
- 快速定位条款。
- 判断答案是否足够支撑报告、方案或审查意见。

### 4.2 专业审核人员

包括资源储量评审、报告审查、方案评审、合规审查相关人员。

核心需求：

- 核验引用是否准确。
- 对比不同标准和不同版本。
- 发现答案证据不足或标准状态异常。

### 4.3 知识库维护人员

负责导入标准、政策、报告模板、内部资料和知识图谱数据。

核心需求：

- 看到系统引用了哪些资料。
- 发现缺失资料和低质量解析结果。
- 维护标准状态、版本和来源可信度。

## 5. 核心使用场景

### 5.1 标准依据查询

用户问题：

```text
哪个标准规定了金矿基本工程间距？
```

系统应返回：

- 可能相关的标准或规范。
- 具体条款或章节。
- 引用片段。
- 是否存在矿种、勘查阶段、勘查类型等适用条件。
- 若当前知识库无条款级证据，应明确说明。

### 5.2 标准正文解释

用户问题：

```text
GB/T 17766-2020 中资源量和储量的分类关系是什么？
```

系统应返回：

- 基于正文或已入库解析文本的解释。
- 对应章节或条款。
- 原文引用或摘要。
- 不确定性说明。

### 5.3 标准状态核验

用户问题：

```text
DZ/T 0321-2018 方解石矿地质勘查规范现在还有效吗？
```

系统应返回：

- 不同官方来源的状态信息。
- 来源 URL。
- 若来源状态冲突，明确提示冲突。
- 不直接给出未经核实的单一结论。

### 5.4 知识库缺失时的联网补充

用户问题命中本地知识库不足时，系统应尝试查询官方标准平台，补充标准号、名称、状态、日期、全文入口等信息。

如果只找到元数据或图片式正文，系统只能说明“找到标准信息，但未获得可用于条款级回答的正文文本”。

如果官方页面提供图片型正文或分页阅读入口，系统可以创建后台 OCR 候选任务。本次回答不等待完整 OCR，也不直接把新识别文本写入后台正式知识库；识别结果先进入候选暂存区，等待管理员批量确认后再正式入库。

## 6. MVP 范围

### 6.1 包含

- 单页面问答入口。
- 基础 API 调用入口，供外部 Agent 和受控客户端复用。
- 标准目录查询入口，用于查询知识库中是否已有某个标准。
- 问题输入和回答展示。
- 本地知识库检索结果接入。
- 答案引用展示。
- 检索依据摘要展示。
- 来源类型和正文可访问状态展示。
- 证据不足提示。
- 用户反馈：有用、无用、引用错误、证据不足。
- 登录用户的服务端会话历史；对“还有哪些文件”“该标准是否还有规定”等依赖上下文的追问，先结合上一轮用户问题补全检索查询，再进入领域判断和检索。
- 邀请码与邮箱验证注册、登录、用户 API Key、每日次数配额和调整记录。
- 领域相关性判断：无关问题直接拒答，不进入检索、联网、OCR 或补库记录。
- 知识库缺口记录：相关问题无证据时，可异步记录为补库候选任务。

### 6.2 不包含

- 完整用户上传与管理员审核流程，作为 V1 功能预留。
- 完整后台知识库管理界面。
- 完整组织/企业权限体系。
- 任何充值、订阅、支付、退款或发票功能。
- 复杂报告生成。
- 全自动标准入库审批。
- 多租户公共网关和复杂组织权限体系。

## 7. 功能需求

### 7.1 问答输入

用户可以在问答页面输入自然语言问题。

要求：

- 支持中文专业问题。
- 支持较长问题，至少 1000 字。
- 支持回车提交或按钮提交。
- 提交后显示加载状态。
- 防止重复提交。

问答入口必须先进行领域相关性判断。

属于服务范围的问题包括：

- 矿产资源、地质勘查、矿山设计、矿业权管理、自然资源管理相关问题。
- 标准、规范、规程、政策、技术要求、条款解释和标准状态核验。
- 与矿产资源技术报告、资源储量、工程间距、勘查类型、控制程度等业务相关的问题。

不属于服务范围的问题包括：

- 通用数学、闲聊、写作、翻译、编程、娱乐等与本产品领域无关的问题。
- 明显试图把系统当作通用大模型调用的问题。
- 恶意提示词注入、要求泄露系统提示词、绕过限制或消耗 token 的请求。

无关问题处理：

- 直接返回固定拒答，例如“本服务仅回答矿产资源标准规范及相关政策技术问题，无法处理该问题”。
- 不调用知识库、联网搜索、OCR、多模态模型或长上下文推理。
- 不记录为知识库缺口或补库任务。
- 可记录最小化的安全计数和调用日志，用于限流和风控。

### 7.2 标准目录查询

用户可以查询知识库中是否已有目标标准。

支持查询条件：

- 标准号。
- 标准名称。
- 标准状态。
- 文档类型。
- 是否有正文。
- 是否 OCR 完成。
- 是否可用于问答。

结果展示：

- 标准名称。
- 标准号。
- 状态。
- 来源。
- 正文可访问状态。
- 是否已解析入库。
- 入库时间。
- 是否可用于问答。

该功能应进入 MVP，因为用户需要先知道库中有没有目标标准。

### 7.3 答案生成

系统根据检索证据生成结构化答案。

答案必须包含：

- 直接结论。
- 依据来源。
- 条款、章节或页码。
- 原文引用或证据摘要。
- 适用条件。
- 不确定性说明。

如果没有条款级证据，答案必须降级为说明性回答，不能生成确定性规范结论。

如果问题属于服务范围但知识库没有可用证据，系统可以返回知识库缺口响应：

- 明确说明当前知识库暂未找到可用于回答的条款级依据。
- 不等待联网搜索、OCR 或人工审核完成。
- 可生成 `knowledge_gap_task`，进入后台低优先级补库队列。
- 允许用户通过任务编号或后续再次查询获取更新结果。

知识库缺口任务不等于答案证据。任务处理完成并经管理员审核入库前，不能作为公共问答依据。

### 7.4 引用来源展示

每条引用至少包含：

- 文件名称。
- 标准号或文档编号。
- 来源类型。
- 正文可访问状态。
- 章节或条款。
- 页码。
- 引用片段。
- 来源 URL 或本地文档 ID。
- 检索得分。

### 7.5 检索依据展示

系统应展示本次回答使用的检索概况：

- 全文搜索命中数量。
- 向量检索命中数量，如已启用。
- 知识图谱命中数量，如已启用。
- 联网搜索命中数量。
- 是否存在条款级证据。
- 是否存在来源冲突。

### 7.5.1 意图理解、术语归一和证据校验

系统不能只依赖关键词检索。对于自然语言问题，应在进入检索和答案生成前做轻量意图理解和术语归一。

第一阶段至少支持以下意图类型：

- `authority_responsibility`: 权限归属、负责机构、哪一级部门办理、谁负责。
- `service_materials`: 矿业权登记、延续、变更等事项需要提交的材料或资料清单。
- `service_procedure_basis`: 矿业权事项的办理依据、办理程序和适用文件。
- `legal_responsibility`: 报告真实性、弄虚作假等法定责任主体。
- `projection_numeric_rule`: 有限/无限外推的比例、距离基准和适用条件。
- `related_documents`: 基于上一轮主题继续查找其他文件或相关规定。
- `standard_selection`: 某矿种、事项或报告应使用哪个标准或规范。
- `numeric_table_lookup`: 工程间距、规模划分、指标值、表格交叉读取。
- `clause_comparison`: 不同标准、不同矿种或不同版本的条款差异。
- `status_verification`: 标准或政策是否现行、是否废止、是否替代。
- `definition_explanation`: 术语定义、分类关系、适用条件解释。
- `out_of_scope`: 与矿产资源、标准规范、自然资源管理无关的问题。

系统应维护领域术语库 `domain_lexicon`，用于把用户口语表达归一为标准表达和检索意图。

权限归属问题必须同时维护三个相互独立的语义角色：

- `license_issuer_level`：当前有效勘查许可证或采矿许可证的颁发层级。
- `mining_right_granting_level`：矿业权出让、配置或登记权限层级。
- `filing_authority`：矿产资源储量评审备案负责机关；现行规则按许可证颁发层级确定。

矿种、矿山规模和矿业权出让权限只能作为背景条件，不能替代许可证颁发机关。确定性角色已经明确时直接检索和回答；角色冲突或表述含混时，允许 DeepSeek 在首次检索前补充结构化角色，但不得覆盖已确定角色或根据矿种、规模猜测发证机关。

示例：

| 用户表达 | 标准表达 | 意图 |
| --- | --- | --- |
| 采矿证 | 采矿许可证、采矿权 | `license_reference`，由问句中的“材料/办理/评审机构”等动作继续确定业务意图 |
| 探矿证 | 勘查许可证、探矿权 | `license_reference`，不能仅凭证件词触发权限路由 |
| 储量报告评审 | 矿产资源储量评审备案 | `authority_responsibility` |
| 去哪个机构、找谁办、谁负责 | 负责部门、权限归属 | `authority_responsibility` |
| 沙金 | 砂金、金属砂矿类 | `standard_selection` |
| 工程距离 | 勘查工程间距 | `numeric_table_lookup` |
| `1类型`、`I类型`、`Ⅰ类型`、`一类型` | `Ⅰ类型` | `numeric_table_lookup` |

检索前的常见写法、口语和 Unicode 数字归一优先采用本地确定性规则。目标明确且已有稳定 Schema 的问题不额外调用规划模型；模糊、比较、跨文件和关系型问题由 DeepSeek 在检索前生成结构化查询计划。计划至少包含规范化问题、意图、文档类型、检索扩展、排除概念、证据关系组和搜索模式，并由系统白名单与默认 Schema 校验。

查询理解采用分层策略：先做领域门控、机械归一和会话追问补全；复杂问题在首次检索前调用模型理解目标关系，不能等到答案生成阶段才介入。第一轮证据不足时，证据审查器可给出一次 refined query，系统最多执行第二轮检索。默认不使用 HyDE 或无边界多查询生成。

确定性识别出的高价值关系意图、用户明确给出的标准号和经过验证的证据条件属于受保护约束。大模型只能补充同义词、专业表达和检索提示，不能覆盖受保护意图或删除证据条件。大模型推测的文件标题仅作为软排序提示，不得直接排除其他文件；软范围证据不足时必须保留全库混合检索能力。

检索性能采用分级预算：普通问题默认返回 10 条、跨标准比较返回 20 条；内部召回按确定性文档范围、普通全库查询、跨标准比较和完整材料清单分别设置预算，不能用统一的大 `top_k`。证据审查模型最多接收 10 条紧凑候选，并只负责选择最多 4 条代表性直接证据，不同时生成最终答案。

结构化查询规划和证据筛选应关闭模型思考模式，默认 `max_tokens` 分别为 600 和 800；最终答案默认 1000。一次请求内的模型阶段复用 HTTP 连接，并只传递回答所需的紧凑检索计划。不得为了降低延迟截断必要的材料清单、表格行列或多标准比较证据。

在线 embedding 优先通过 ANN 索引检索。ANN 元数据一致性校验可短时缓存；ANN 或 embedding 不可用时，精确 JSON 向量和本地 N-gram 哈希向量只允许在 SQL 已缩小范围且候选不超过 100 条时扫描，超限时依赖 FTS、知识图谱和异步补库，不进行大范围逐条余弦计算。

`chunk_embeddings` 使用 `(chunk_id, vector_model)` 复合主键，其中 `chunk_id` 为首列，已满足按 chunk 回表的索引需求，不应再增加重复索引。知识图谱模糊检索只针对标准、矿种、机构、材料、表格等命名实体，条款正文由 FTS/ANN 负责；KG 连接应先物化小实体候选集，再连接关系和 chunk。

高价值关系问题采用证据槽位判断充分性。例如探矿权转采矿权问题至少区分经评审备案报告、勘查程度条件和报告类型限制；伴生矿产资源量类型问题至少区分基本分析且研究充分、基本分析但研究不足、仅组合分析三种情形。证据槽位已经由条款满足时，大模型不得因原文没有重复用户口语表达而判定证据不足。

对于“矿种 + 勘查类型 + 工程间距”等目标明确的问题，应先利用知识库 schema 和标准元数据锁定一个或少量候选标准，再在候选文档内检索条款或表格；候选不存在、证据不满足意图或路由置信度不足时，必须自动退回全库检索。对于“哪些规范存在不一致”等跨标准比较问题，不得使用单标准限定，应保留全量检索。

对于高价值意图，答案生成前必须做证据可回答性校验：

- 权限归属类问题必须命中包含负责主体的证据，例如“自然资源部负责”“省级自然资源主管部门负责”。
- 办证材料类问题必须命中对应事项和申请类型的材料表/附件；只命中“按附件执行”时只能说明适用依据，不得推测逐项材料。
- 办理依据类问题必须命中办理事项对应的现行文件或官方办事指南，不能被评审备案权限文件替代。
- 报告真实性责任必须命中“真实性负责/不得弄虚作假”等直接责任条款，不能按许可证颁发层级回答。
- 外推比例类问题必须同时命中外推类型、比例和距离基准；`1/2尖推` 与 `1/4平推` 不得脱离有限/无限外推条件混用。
- 标准适用类问题必须命中标准标题、标准号或目录证据，不能只根据训练数据推测。
- 表格数值类问题必须命中表格或等价结构化证据，并按行列交叉读取。
- 用户明确要求表格时，应优先使用结构化表格 chunk；命中“见表 E.1 至表 E.5”等引用关系后，应展开同一文件中的对应表格，并以可渲染的 Markdown 表格返回。
- 条款差异类问题必须至少命中两个可比较的条款或明确说明只能列出单一来源。
- 如果命中证据与问题意图不一致，例如用油气标准回答金矿储量报告评审机构，应降级为证据不足或重新检索，不能让模型自由解释。

后续增强方向：

- 由 KB 维护可版本化的 `domain_lexicon`，问答后端只消费其归一化结果。
- 对高价值政策条款抽取 `部门 -> 负责 -> 事项`、`标准 -> 适用 -> 矿种/事项`、`文件 -> 规定 -> 权限` 等知识图谱关系。
- v1.0 已在全文、向量和图谱召回之后接入 DeepSeek 结构化证据审查，用于判断候选是否直接表达目标关系；embedding 只负责召回，不能决定证据是否可回答。
- 检索采用按需级联：简单且目标明确的问题可在 Schema + FTS/KG 已充分时跳过向量；跨标准比较和复杂关系问题使用 USEARCH ANN 补充语义召回。
- ANN 不可用时只允许在已锁定候选文档的范围内精确计算，不得恢复全库逐条 JSON 向量扫描。
- 复杂问题的回答草稿与证据审查可在同一次结构化模型调用中完成；模型输出仍须经过证据索引和系统关系约束验证。

### 7.5.2 候选排序与证据多样性（v1.0.6 已受控实施）

优先级：普通。

v1.0.6 已接入意图感知 MMR（Maximal Marginal Relevance），但不进行全局强制重排。只有符合适用意图、没有硬 Schema 范围或用户指定标准，且前 5 条至少 4 条来自同一文档时才触发；否则保持现有 RRF/规则排序。

MMR 不得作为全局统一排序规则。适用范围包括：

- 跨标准、跨制度的条款比较和代表性差异列举。
- “还有哪些文件规定了相关内容”等扩展检索。
- 普通综合政策查询和需要多个独立来源支撑的回答。
- 正确证据已经进入候选集，但被同一文档或高度相似 chunk 挤出前列的情况。

以下意图默认禁用 MMR，或等价使用 `lambda=1.0` 保留纯相关度排序：

- 指定标准、指定条款和标准目录精确查询。
- 工程间距、工业指标等结构化表格行列读取。
- 附件材料清单、办事指南材料目录等需要同一文档多个 chunk 的完整性任务。
- 高价值证据槽位已经明确，需要组合同一文件多个条款才能完成回答的任务。

计划排序位置为：Schema/FTS/KG/ANN 混合召回 -> 硬范围和证据槽位校验 -> 保留强制证据 -> 对剩余候选执行 MMR -> 证据审查/答案生成。MMR 只能重排已经召回的候选，不能替代问题转写、Schema 路由、召回扩展或知识库补齐。

参数策略：

- 符合条件但暂无历史数据时，从 `lambda=0.7` 开始。
- 比较类问题或前 5 条中至少 4 条来自同一文档/高度相似内容时，可降低到 `0.55～0.65`。
- 候选已经较分散、需要优先保持相关度时，可提高到 `0.8～0.9`。
- “结果相关度不够”只有在正确证据已经进入候选池时才可通过提高 lambda 改善排序；如果正确证据根本未召回，应回到查询理解和召回层处理。

持续评估门槛：

- v1.0.6 已建立 25 题基准并保留候选、预期来源和检索策略结果；后续继续加入真实用户反馈。
- 统计前 5 条候选的同文档比例、语义重复率、标准覆盖数和正确证据是否被挤出。
- 当前 25 题基准没有真实问题触发重复阈值，因此 MMR 保持按请求动态触发，不扩大到全局查询。

实现约束与验收标准：

- 仅对每次检索已经形成的 30～80 条候选计算多样性，不得触发全库向量扫描或新增远程 embedding 调用。
- 强制证据槽位、用户明确指定的标准和确定性 Schema 结果不得被 MMR 删除。
- 使用功能开关和按意图配置，支持快速关闭和对照测试。
- 比较/综合查询的代表性标准覆盖率提高，前 5 条语义重复率下降。
- 精确条款、表格和材料清单回归结果不得下降。
- MMR 层新增 P95 延迟目标不超过 100 ms。

### 7.5.3 查询改写、多查询扩展与模型采样策略

优先级：普通。

#### 查询改写（已采用，继续增强）

受控 Query Rewriting 适合 geowiki，应继续作为查询理解层的基础能力，但不能把所有问题都交给大模型自由改写。

- 简单、目标明确和已有稳定 Schema 的问题继续使用本地机械归一、领域词典和确定性意图，不调用查询规划模型。
- 模糊、口语化、跨文件、比较和关系型问题允许由 DeepSeek 输出一条结构化 `canonical_query`，同时给出主题、必需术语、同义术语、排除术语、文档类型和证据关系组。
- 改写必须保留用户明确提供的标准号、文号、条款号、矿种、数值、比例、勘查阶段、许可证颁发主体和适用条件。
- 查询规划模型只能生成检索计划，不能把模型记忆中的答案或推测文件当作证据。
- 第一轮证据不足时，证据审查器最多生成一条 `refined_query`，系统最多执行第二轮受控检索。
- 检索轨迹应同时保存原问题、规范化问题、最终检索问题、扩展术语、是否触发第二轮以及第二轮新增的直接证据，供后续评估改写是否真正提高召回。

查询改写的验收重点不是“文本更专业”，而是正确证据召回率提高，同时不得改变用户的真实问题或扩大到无关标准。

#### Multi-Query（v1.0.6 已受控实施）

Multi-Query 不全局启用，仅作为复杂问题首轮证据不足后的受控召回增强手段，且优先级低于单查询改写和证据缺口分析。v1.0.6 最多并发执行 2 条补充查询，其中优先保留证据审查器生成的一条 `refined_query`，剩余额度用于绑定独立证据目标的模型子查询。

允许考虑启用的场景：

- 跨标准比较，需要分别检索各比较维度或各类距离基准。
- “还有哪些文件规定了相关内容”等扩展查询。
- 一个问题同时包含政策条件、技术标准限制和办理材料等多个独立证据槽位。
- 单条规范化查询已执行，但第一轮未覆盖全部必需证据组。

默认禁用的场景：

- 标准号、文号、条款号明确的精确查询。
- 工程间距、工业指标和分类表等结构化表格读取。
- 附件材料清单和办事指南完整材料目录。
- 当前 Schema/FTS/KG 已经找到充分直接证据的简单问题。

计划实现约束：

- 每次最多生成 2～3 条具有不同证据目标的子查询，不能只生成同义改写后重复搜索。
- 每条子查询必须绑定明确的证据槽位或比较维度，例如“行政许可条件”“报告类型限制”“外推距离基准”。
- 用户明确的标准号、矿种、时间范围和受保护意图必须继承到每条子查询。
- 子查询结果使用现有 RRF、规则评分和证据校验合并，并按 `chunk_id`、文档和语义重复去重。
- 第一条查询已满足全部证据槽位时立即停止，不执行额外子查询。
- Multi-Query 只负责提高召回，不直接生成答案；不得引入 HyDE 文本作为证据。
- 结构化子查询生成使用 `temperature=0`、关闭思考模式并限制输出长度。
- 实施前必须通过离线评估证明新增查询带来了新的直接证据，而不是只增加重复候选、模型调用和延迟。

#### Temperature（采用当前低随机性策略）

当前温度策略适合专业标准问答，应在 PRD 中固定为默认基线：

- 查询规划、查询改写、Multi-Query 生成和证据审查等结构化 JSON 调用使用 `temperature=0`。
- 最终自然语言回答默认使用 `temperature=0.2`，允许配置在 `0～0.3` 范围内，不应为了语言多样性提高到高随机区间。
- 温度参数后续可以环境变量化，但生产默认值和允许范围必须由后端控制，不能由普通 API 用户任意指定。
- 温度调整只影响生成稳定性，不得作为解决检索缺失、证据错误或来源重复的手段。

#### USEARCH expansion_search（保留当前基线，离线调优）

v1.0.6 ANN 基线为 `connectivity=24`、`expansion_add=128`、运行时 `expansion_search=64`。其中 `expansion_search` 是 USEARCH/HNSW 的图搜索广度参数，不属于查询改写、多查询扩展或 MMR。

- 25 题、22,778 向量离线基准比较了 `64/96/128`；三者最低 Recall@20 均为 `0.95`，平均 Recall@20 分别为 `0.994/0.996/0.996`，P95 分别为 `0.518/0.605/0.646 ms`。
- 当前选择 `64` 作为运行时及后续建索引基线；参数仍可通过环境变量和离线工具重新评估。
- 如果提高搜索广度没有带来可测量的正确证据提升，应保留较低成本参数。
- ANN 参数调整必须与索引版本、embedding 模型和知识库快照一同记录并重新发布。

### 7.5.4 分阶段 Prompt 校准层（低优先级）

优先级：低。

Prompt 校准层用于把已经验证的领域理解规则、正反例和输出约束按检索阶段与意图动态注入模型，以较低成本改善第三方大模型在固定业务模式下的稳定性。它只能模拟部分 LoRA 行为，不修改模型权重，也不能替代知识库、Schema、召回、Chunk 治理和证据校验。

适合校准的阶段：

- 查询规划：区分背景条件与真正判断依据，例如矿山规模与储量评审权限、详查程度与报告名称。
- 证据审查：区分直接证据和只包含相同关键词的旁证，例如父政策附件指引与附件正文、普通工程间距表与外推距离规则。
- 最终回答：统一短引用、适用条件、来源展示、证据不足表述和比较类输出格式。

不适合交给 Prompt 校准解决的问题：

- 知识库未收录目标文件或文件状态错误。
- Chunk、表格、页码和附件关系解析错误。
- Schema 范围、FTS/KG/ANN 召回不足或正确证据根本未进入候选集。
- 数据库、embedding、ANN 和模型网络调用的性能问题。

计划实现方式：

- 建立版本化 Prompt Registry，按阶段和意图保存配置，例如 `planner/authority/v1`、`reranker/projection/v1`、`answer/comparison/v1`。
- 每次只注入当前意图需要的短规则以及 1～3 个经过人工确认的正例/反例，禁止形成不断膨胀的全局 Prompt。
- 受保护意图、标准号、文号、数值、证据槽位和系统安全规则继续由代码校验，Prompt 无权覆盖。
- 查询规划和证据审查继续使用 `temperature=0`、关闭思考模式和结构化输出。
- 所有 Prompt 版本必须支持功能开关、灰度测试、回滚和检索轨迹追踪。
- Prompt 中不得嵌入未获授权的标准全文；示例优先使用抽象化、短引文或公开许可内容。

实施门槛与验收：

- 至少积累 30～50 个已经明确属于“模型看到正确证据但理解、筛选或表达错误”的失败案例。
- 按查询规划、证据审查和答案表达分别建立校准集，不能把检索缺失案例混入 Prompt 效果评估。
- 与现有 Prompt 做离线和灰度对照，统计意图准确率、直接证据选择准确率、引用错误率、答案稳定性、token 和延迟变化。
- 只有在目标错误类型上有稳定提升，且不降低现有高价值回归集准确率时才上线。
- Prompt 校准属于低优先级增强，不得早于知识库缺口、系统性召回错误、材料/表格完整性和普通优先级排序优化。

### 7.6 联网补充

联网能力只作为本地知识库不足时的补充。

默认交互策略是“快速返回、后台补库”。用户请求不等待完整联网搜索、OCR 或人工审核。系统可以先返回证据不足，并将真实业务问题记录为知识库缺口任务，后续由后台低优先级队列慢慢处理。

系统可以联网查询：

- 全国标准信息公共服务平台。
- 国家标准全文公开系统。
- 自然资源标准化信息服务平台。
- 其他后续确认的官方来源。

联网补充可用于：

- 标准号、标准名称、状态、发布日期、实施日期。
- 主管部门、归口单位、备案信息。
- 是否存在官方全文阅读入口。
- 候选标准发现。

联网补充不能直接用于：

- 在没有正文文本的情况下回答条款级问题。
- 将第三方文档当作官方正文。
- 忽略不同来源之间的状态冲突。

### 7.6.1 异步补库任务

当问题通过领域相关性判断，但正式知识库没有条款级证据时，系统可以创建异步补库任务。

任务用途：

- 收集真实用户需求。
- 统计高频缺失标准和高频问题。
- 后台排队执行官方来源搜索、OCR、解析和候选暂存。
- 管理员批量审核后统一入库，供后续复用。

任务不要求并发执行。MVP 阶段可以采用单 worker 或定时脚本，例如每小时处理少量任务，以降低服务器压力。

任务建议字段：

- task_id
- question
- detected_domain
- detected_keywords
- candidate_standard_no
- candidate_title
- status
- priority
- created_at
- processed_at
- result_candidate_id

任务状态建议：

- `queued`: 已记录，等待处理。
- `searching`: 正在搜索官方来源。
- `ocr_pending`: 已找到图片型或视觉型来源，等待 OCR。
- `ocr_running`: OCR 处理中。
- `review_pending`: OCR 或解析结果等待管理员审核。
- `approved_for_kb`: 审核通过，准备或已经正式入库。
- `rejected`: 不适合入库或来源不可用。
- `done`: 任务处理结束。

无关问题不能创建补库任务。

### 7.7 来源可访问性分类

系统必须为每个来源标记 `source_type`：

- `official_metadata`: 官方元数据。
- `official_fulltext`: 官方正文可文本抽取。
- `official_visual`: 官方正文存在，但为图片、切片或 canvas/PDF.js 渲染。
- `third_party_candidate`: 第三方候选文档。
- `local_kb`: 本地知识库文档。
- `unavailable`: 未找到可用来源。

系统必须为每个来源标记 `text_access`：

- `metadata_only`
- `html_text`
- `pdf_text`
- `image_ocr_required`
- `ocr_text`
- `unavailable`

### 7.8 来源冲突提示

当不同来源对标准状态、发布日期、实施日期、废止状态或替代关系给出不同信息时，系统必须提示冲突。

示例：

```text
该标准状态存在来源差异：SAMR 行业标准页面显示废止；自然资源标准化信息服务平台详情页显示现行标签，但时间轴包含废止节点且未给出废止日期。建议以主管部门最新公告或正式出版信息进一步核验。
```

状态优先级：

- 国家标准优先采用国家标准全文公开系统和全国标准信息公共服务平台。
- 自然资源行业标准优先采用自然资源标准化信息服务平台。
- 两个官方来源冲突时，保留冲突信息并提示用户，不静默覆盖。

### 7.9 用户上传与管理员审核

用户上传资料作为 V1 功能预留。

规则：

- 云端内测已经要求邀请码登录；上传功能继续复用现有账号体系。
- 用户上传资料默认只对上传用户可见。
- 用户可申请将资料提交管理员审核。
- 管理员审核通过后，资料才能进入受控服务可检索范围。
- 审核未通过的资料不能影响其他用户，也不能作为服务端共享答案依据。
- 用户提问触发的联网 OCR 或网页解析结果也应进入候选暂存区，不能自动进入后台正式知识库。

审核内容：

- 来源是否可靠。
- 标准号、名称、版本是否正确。
- 是否重复入库。
- OCR 或解析质量是否达标。
- 是否存在版权或使用边界问题。

候选暂存区要求：

- 保存触发问题、候选标准、官方来源 URL、采集时间和处理状态。
- 保存 OCR 文本、页码映射、置信度、OCR 工具和版本。
- 支持管理员按标准号、来源、置信度和创建时间筛选。
- 支持批量通过、拒绝、要求重新 OCR 或标记仅供内部使用。
- 审核通过后才写入正式知识库、全文索引、向量索引或图谱。

### 7.10 用户反馈

用户可以对答案反馈：

- 有用。
- 无用。
- 引用错误。
- 标准状态疑似错误。
- 没有回答到问题。
- 证据不足。

反馈用于后续改进检索、知识库和提示词。

反馈必须写入可查询的处理队列并关联 `request_id`、会话和原问题：

- `quote_too_long`、`answer_too_vague`、`format_issue` 等进入程序/问答优化通道，可由主开发 Agent 定位并修复。
- `wrong_standard`、`wrong_clause`、`missing_evidence` 等进入知识库审核通道，由管理员或 KB 专员确认后才能修改正式知识库。
- `other` 进入人工判断通道；满意反馈直接关闭。
- 管理员后台可以查看问题、用户、原因、处理通道和状态，并记录处理中、待 KB 审核、已解决或已忽略及处理说明。

## 8. 答案策略

### 8.1 本地知识库优先

系统首先使用本地知识库中的已解析、已校验文档。

本地知识库命中条款级证据时，可以生成规范性回答。

### 8.2 联网只补充，不兜底编造

联网搜索只补充来源和元数据。若联网只找到元数据，不生成条款级结论。

### 8.3 OCR 结果需标记

如果正文来自 OCR 或视觉解析，应保留解析来源和置信度。

低置信度 OCR 文本不应作为强结论依据。

推荐 OCR 备选工具：

- PaddleOCR：优先作为中文标准、扫描 PDF、图片型 PDF 的 OCR 试验工具。
- PP-StructureV3：用于版面分析、章节结构识别、表格识别和复杂文档解析试验。
- TableRecognitionPipelineV2：用于表格较多的标准附录、指标表、分类表识别试验。

选型理由：

- 对中文识别支持较好。
- 支持文档结构解析和表格识别，适合标准、规范、规程类文件。
- 已在本地 Codex 专用虚拟环境中安装验证，可作为 MVP 后续 OCR 试验基础。

使用限制：

- OCR 结果必须保存页码、来源文件、模型版本和置信度。
- OCR 结果进入知识库前应抽样校验。
- 图片型标准全文的抓取、长期保存和对外展示需要先确认版权与使用边界。

### 8.4 第三方资料谨慎使用

第三方 PDF、网页、文库类页面只能作为候选来源。

除非经过校验，否则不能作为最终标准正文依据。

用户上传的第三方 PDF 默认仅对上传用户开放。经管理员审核通过后，才能进入受控服务可检索范围。

## 9. 数据来源策略

### 9.1 官方国标来源

国家标准可优先查询：

- `std.samr.gov.cn`
- `openstd.samr.gov.cn`

已验证问题：

- 部分国标在线预览为图片切片或视觉阅读器。
- 页面可提供元数据，但正文不一定能被文本模型直接读取。

### 9.2 官方行标来源

行业标准可优先查询：

- `std.samr.gov.cn/hb`
- `nrsis.org.cn`

已验证问题：

- `std.samr.gov.cn/hb` 可能只提供元数据。
- `nrsis.org.cn` 对自然资源领域行标可能提供全文阅读入口。
- `nrsis.org.cn` 阅读器可能按页返回图片型 PDF，需要 OCR。

### 9.3 本地知识库

本地知识库应保存：

- 原始文件。
- 解析文本。
- OCR 文本。
- 页码映射。
- 条款层级。
- 标准状态和版本。
- 来源 URL。
- 入库时间。
- 校验状态。

## 10. 知识库构建需求

知识库由独立服务或另一个 agent 负责构建。本产品后端只依赖知识库服务接口，不直接绑定其内部实现。

详细要求见 [KNOWLEDGE_BASE_REQUIREMENTS.md](/home/nalanmading/My-project/my-1st-agent/docs/KNOWLEDGE_BASE_REQUIREMENTS.md)。

PRD 层面对知识库的要求是：

- 知识库必须先建立稳定 schema，再做索引和检索。
- 知识库必须返回证据列表，不直接生成最终答案。
- 证据必须可追溯到文档、章节、条款或页码。
- MVP 阶段优先支持标准目录查询和 Elasticsearch 全文搜索。
- v1.0 使用 `text-embedding-v4` + USEARCH ANN 和 SQLite KG；ChromaDB/专用向量数据库、Neo4j 作为规模化替换选项。
- Embedding Provider 必须可配置，不绑定 `deepseek-v4-flash`。
- 知识库需要标记来源类型、正文可访问状态和质量状态。
- 当本地证据不足时，知识库应返回 `needs_web_supplement: true`，由问答后端或联网模块补齐官方元数据、全文入口或 OCR 任务。

### 10.1 接受的知识库架构

软件应明确声明支持的知识库架构。这样开源本地版、云端受控测试版和机构内部部署版都可以按同一套接口接入，agent 也能最大化利用标准号、条款、页码、来源、OCR 置信度和审核状态。

最低兼容架构：

- 结构化文档元数据。
- 条款或文本片段表。
- 页码或来源定位字段。
- 标准目录查询接口。
- 证据检索接口。
- 来源类型、正文可访问状态和质量状态字段。
- 领域术语库或同义词表，用于用户表达、标准术语和意图标签之间的映射。

推荐 MVP 架构：

- PostgreSQL 或等价关系型数据库：保存 schema、标准元数据、条款、页码、表格、上传和审核状态。
- Elasticsearch：提供标准号、术语、条文关键词等全文搜索。
- 对象存储或本地文件目录：保存用户上传文件、原始 PDF、OCR 中间结果，按授权策略决定是否长期保存。
- 后台任务队列：处理 OCR、解析、入库、质量检查。
- 统一知识库 API：至少提供 `/knowledge/search` 和 `/knowledge/standards`。
- `domain_lexicon`：保存同义词、规范术语、意图标签、适用场景和负面降权规则。

增强架构：

- USEARCH：v1.0 的私有 ANN 语义召回索引；后续可按规模替换为 ChromaDB、Milvus、pgvector 或等价服务。
- Embedding Provider：独立配置，不使用 `deepseek-v4-flash`。当前使用阿里云百炼/DashScope `text-embedding-v4`。
- SQLite KG：v1.0 已用于标准、矿种、条款、材料、时限和责任关系；Neo4j 是后续规模化选项。
- 按需混合检索：全文搜索、结构化字段和知识图谱优先；证据不足时再补充向量召回并统一排序。
- Reranker：用于证据重排序和可回答性判断，优先判断“证据是否回答了用户意图”，而不是只看语义相似度。

如果用户已有自己的知识库，只要能通过适配器返回本系统要求的证据格式，也可以接入；但缺少 schema、条款定位、页码映射和质量状态时，agent 只能降级回答。

## 11. 非功能需求

### 11.1 准确性

专业答案必须以证据为核心。系统宁可回答“不足以判断”，也不能编造条款。

### 11.2 可追溯性

每个结论应尽量追溯到文档、页码、章节或条款。

### 11.3 响应速度

v1.0 目标：

- 已验证的结构化常规问题 2 秒内返回。
- 需要模型规划和证据审查的复杂比较问题目标 30 秒内返回。
- 需要联网补充的问题 30 秒内返回初步结果。
- OCR 不要求同步完成，可进入后台处理。

### 11.4 安全与合规

- 不提交 `.env` 或 API Key。
- 不公开知识库本体、标准全文库、OCR 全文、索引、数据库或 `/knowledge/*` 后台检索服务。
- 尊重官方平台的使用提示和版权声明。
- 对外展示内容以必要、限长、可核验的引用片段、摘要和来源说明为主。
- 公开社区版不得包含真实标准全文、OCR 后全文、预构建真实标准知识库或向量库。
- 机构内部部署版可按资料授权范围配置长文本返回、全文索引和内部导出。
- 云端受控测试版只开放问答 API 输出，不提供知识库下载、镜像、全文下载、批量导出、大段原文展示或直接知识库检索入口。
- 公共 API 必须具备 API Key、每日配额、限流、调用日志和用量统计。
- 限流优先采用 Redis 作为共享计数后端，支持多进程和未来多节点部署；Redis 不可用时应降级为本地内存限流，避免开发和单机内测被外部依赖阻断。
- 无关问题必须在低成本阶段拒答，避免被当作通用大模型接口消耗 token。
- 恶意调用、异常高频调用、提示词注入和批量探测请求应进入风控统计，必要时限流或封禁 API Key。

### 11.5 成本控制

系统应把成本高的操作放在后置流程。

要求：

- 先做领域相关性判断，再进入检索和模型推理。
- 先查正式知识库，再决定是否联网补充。
- OCR、多模态理解、长上下文推理默认异步执行。
- 知识库缺口任务可排队慢处理，不要求并发。
- 后台补库优先处理高频、明确、领域相关的问题。
- API 返回应包含配额消费和剩余次数，便于用量控制、审计和防滥用。

### 11.6 可维护性

- 检索服务、联网补充服务、OCR 服务和 LLM 生成服务应解耦。
- 来源分类、答案策略和提示词应可配置。
- 后续可以替换模型供应商。

## 12. 部署与开放形态

产品支持三种部署形态，详见 [LICENSING_AND_REPOSITORIES.md](/home/nalanmading/My-project/my-1st-agent/docs/LICENSING_AND_REPOSITORIES.md)。

云端受控测试版采用 API-first 形态。前端主要作为内测入口、API 调试台和管理员后台，不设置充值、订单、套餐或其他盈利功能。

知识库作为产品护城河，在任何部署形态下都不对外公开。云端受控测试版只对外提供受控问答 API、任务状态和必要来源定位。

- 不提供官方标准全文公共库或任何形式的知识库本体公开访问。
- 不提供大段原文展示。
- 不提供全文下载或批量导出。
- 用户上传资料默认私有。
- 服务可检索资料必须经过审核或授权。

受控 API 能力：

- 邀请码注册、账号登录和浏览器会话。
- API Key 管理。
- 网页问答与 API 调用共用账号每日次数配额。
- 调用限流，优先 Redis，支持本地内存兜底。
- 用量统计。
- 调用日志和审计。
- 结构化错误码。
- OpenAPI 文档和 SDK 示例。
- 任务状态查询，例如知识库缺口任务、OCR 任务和候选入库任务。

每日配额规则：

- 新用户默认每天可提交 10 次问答，日期按 `Asia/Shanghai` 计算。
- 网页会话和同一账号创建的全部 API Key 共享配额。
- `answered`、`insufficient_evidence` 和 `queued_for_enrichment` 消费 1 次。
- `out_of_scope` 不消费次数，并且不得调用知识库、联网、OCR 或长模型推理；仍受账号、IP/API Key 的分钟级限流和审计约束，防止无成本滥用。
- `system_error` 退回预留次数。
- 管理员可以修改用户长期日上限，也可以给指定日期增加额外次数；所有调整记录操作人、原因和时间。
- 配额必须由后端事务原子预留和结算，前端只展示后端返回的状态。

## 13. 成功指标

MVP 验收指标：

- 用户能提交问题并获得结构化答案。
- 至少展示 1 条来源或明确说明无来源。
- 条款级回答必须包含引用。
- 无证据时不生成确定性结论。
- 用户反馈可被记录、分类并在管理员队列中闭环处理。
- 标准状态冲突可被提示。
- 无关问题能被拒答，且不进入补库任务。
- 知识库缺口问题能生成异步补库任务。

运营观察指标：

- 问题成功回答率。
- 有条款级引用的答案比例。
- 用户反馈“有用”的比例。
- 引用错误反馈比例。
- 证据不足问题 Top N。
- 本地知识库缺失标准 Top N。
- 无关问题拦截数量和比例。
- 知识库缺口任务完成率。
- OCR 候选审核通过率。
- API 调用量、平均成本和平均响应时间。
- 按意图统计前 5 条候选的同文档比例和语义重复率。
- 正确证据已召回但未进入最终引用的比例。
- 查询改写前后直接证据命中率，以及第二轮 `refined_query` 的证据补全率。
- Multi-Query 每增加一次子查询所带来的新增直接证据数、重复候选数、模型成本和延迟。

- 对比原问题、单查询改写、一次 refined query 和 2～3 条受控 Multi-Query 的召回差异。
- 评估 Multi-Query 是否提供新的证据槽位，而不是仅增加同义重复结果。
- 评估当前 RRF/规则排序与意图感知 MMR 的离线差异。
- 比较 USEARCH `expansion_search=64/96/128` 的 Recall@K 与 P95 延迟，保留可验证的最优基线。
- 已以功能开关和按请求重复率阈值接入 MMR；真实基准未触发时不执行。
- 已以按意图功能开关接入 Multi-Query，仅在首轮证据不足后执行，最多 2 条补充查询。
- 回归验证精确条款、表格和材料清单不受影响。

### M11: Prompt 校准层评估

优先级：低。

- 从用户反馈和检索轨迹中筛选 30～50 个“正确证据已存在但模型理解或筛选错误”的案例。
- 分别建立查询规划、证据审查和答案表达校准集。
- 建立版本化 Prompt Registry、按意图加载机制、功能开关和回滚能力。
- 对比现有 Prompt 与校准 Prompt 的准确率、稳定性、token 消耗和 P95 延迟。
- 仅在普通优先级检索优化稳定后进入灰度，不以 Prompt 掩盖知识库或召回问题。

## 16. 已确认产品决策

- 产品显示名为 `geowiki`，中文副标题为“一款专注地质领域的百科全搜”。现有 Python 包名、部署路径和 systemd 单元暂时保留，避免无关迁移风险。
- 云端内测采用邀请码、邮箱验证码注册和登录，不开放匿名问答；网页和用户 API Key 共用账号每日次数。
- 登录用户的会话历史保存在独立应用数据库中，用户可以删除会话；知识库数据库不保存用户账号和配额数据。
- 新用户默认每日 10 次；所有正常完成状态消费 1 次，只有系统异常退回。系统不提供充值、订单或支付功能。
- 引用展示优先包含标准名称、标准号、具体条款和条款原文；页码作为可选核验字段保留。
- v1.0 知识库已完成 schema、标准目录、FTS5、SQLite KG、`text-embedding-v4` 与 USEARCH ANN；索引更新必须与私有 SQLite KB 同步发布。
- 生成模型使用 `deepseek-v4-flash`；embedding 使用独立 provider，不使用 `deepseek-v4-flash`。默认配置为 `EMBEDDING_PROVIDER=aliyun`、`EMBEDDING_MODEL=text-embedding-v4`、`EMBEDDING_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`。
- 在线 embedding 只在文档入库、补库或批量重建时产生模型服务商调用成本；普通问答只需对用户问题做一次 embedding。批量重建真实知识库向量前需要人工确认成本和批量范围。
- 普通问答并非固定调用 embedding：候选标准已锁定且全文/图谱证据足够时应直接跳过；跨标准比较或证据不足时才生成查询向量。
- 官方图片式预览允许 OCR 入库，但来源限定为已确认官方平台，优先是国家标准全文公开系统、全国标准信息公共服务平台和自然资源标准化信息服务平台。
- OCR 文本进入知识库后需要长期保存 OCR 文本、页码映射、置信度、OCR 工具和版本；原始文件是否长期保存按授权和存储策略决定。
- 用户上传第三方 PDF 默认只对上传用户可见，经管理员审核通过后才能进入受控服务可检索范围。
- 标准状态冲突时，国标优先采用国家标准官方平台，自然资源行业标准优先采用自然资源标准化信息服务平台，同时保留冲突提示。
- 社区版继续采用 AGPL-3.0。此前规划的商业许可证路线当前暂停，不属于云端受控测试版功能。
- 私有仓库可继续用于保存不公开的内部部署代码和配置，但不得包含知识库明文或密钥。
