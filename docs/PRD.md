# PRD: geowiki

版本：v2.2.3
日期：2026-07-13
状态：v2.2.3 已完成

## 1. 产品概述

geowiki 是一款专注地质领域的百科全搜，面向矿产资源、地质勘查、矿山设计和自然资源管理场景提供专业知识问答。

用户通过自然语言提出专业问题，系统基于本地知识库、全文搜索、向量检索、知识图谱和有限联网补充能力，返回有依据、可追溯、可核验的答案。

第一阶段产品目标不是做通用聊天机器人，而是解决“专业问题应该依据哪个标准、哪一条、适用于什么条件”的检索和解释问题。

## 2. 背景与问题

矿产资源领域的标准、规范、规程、政策文件数量多，更新周期长，分布来源复杂。实际工作中，技术人员经常需要回答类似问题：

- 某个矿种资源量估算应依据哪个规范？
- 某个术语在标准中如何定义？
- 某类工程间距、勘查类型、控制程度要求在哪个条款中规定？
- 某个标准是否现行，是否存在替代或修改单？

当前主要痛点：

- 标准文件分散，人工检索耗时。
- 关键词检索命中多，但难以快速定位条款。
- 大模型容易在没有证据时生成看似合理但不可核验的答案。
- 官方网站常常只提供元数据，正文可能是图片式预览，文本模型无法直接读取。
- 行标、国标、地方标准和第三方文档之间存在版本、状态和来源可信度差异。

## 3. 产品目标

MVP 阶段目标：

- 提供一个 API-first 的专业问答能力，前端主要用于内测、演示和管理。
- 对每个答案展示引用来源、条款或章节、页码、原文片段或证据摘要。
- 在证据不足时明确说明不足，不编造标准条款。
- 支持本地知识库优先，联网搜索作为补充。
- 为后续接入标准库、OCR、知识图谱、配额统计和开发者 API 打好接口基础。
- 支持开源本地版、云端受控测试版和机构内部部署版三种演进方向。

非目标：

- 不做通用办公聊天助手。
- 不回答与矿产资源、地质勘查、矿山设计、自然资源管理、标准规范或相关政策无关的问题。
- 不替代正式标准出版物和专业人员审查。
- 不在 MVP 阶段做完整知识库后台管理。
- 不在 MVP 阶段做多租户、付费、复杂审批流。
- 不在开源仓库中内置真实标准全文、OCR 后全文或预构建真实标准知识库。

## 4. 目标用户

### 4.1 一线技术人员

包括地勘单位、矿山企业、设计院和咨询单位技术人员。

核心需求：

- 快速找到依据标准。
- 快速定位条款。
- 判断答案是否足够支撑报告、方案或审查意见。

### 4.2 专业审核人员

包括资源储量评审、报告审查、方案评审、合规审查相关人员。

核心需求：

- 核验引用是否准确。
- 对比不同标准和不同版本。
- 发现答案证据不足或标准状态异常。

### 4.3 知识库维护人员

负责导入标准、政策、报告模板、内部资料和知识图谱数据。

核心需求：

- 看到系统引用了哪些资料。
- 发现缺失资料和低质量解析结果。
- 维护标准状态、版本和来源可信度。

## 5. 核心使用场景

### 5.1 标准依据查询

用户问题：

```text
哪个标准规定了金矿基本工程间距？
```

系统应返回：

- 可能相关的标准或规范。
- 具体条款或章节。
- 引用片段。
- 是否存在矿种、勘查阶段、勘查类型等适用条件。
- 若当前知识库无条款级证据，应明确说明。

### 5.2 标准正文解释

用户问题：

```text
GB/T 17766-2020 中资源量和储量的分类关系是什么？
```

系统应返回：

- 基于正文或已入库解析文本的解释。
- 对应章节或条款。
- 原文引用或摘要。
- 不确定性说明。

### 5.3 标准状态核验

用户问题：

```text
DZ/T 0321-2018 方解石矿地质勘查规范现在还有效吗？
```

系统应返回：

- 不同官方来源的状态信息。
- 来源 URL。
- 若来源状态冲突，明确提示冲突。
- 不直接给出未经核实的单一结论。

### 5.4 知识库缺失时的联网补充

用户问题命中本地知识库不足时，系统应尝试查询官方标准平台，补充标准号、名称、状态、日期、全文入口等信息。

如果只找到元数据或图片式正文，系统只能说明“找到标准信息，但未获得可用于条款级回答的正文文本”。

如果官方页面提供图片型正文或分页阅读入口，系统可以创建后台 OCR 候选任务。本次回答不等待完整 OCR，也不直接把新识别文本写入后台正式知识库；识别结果先进入候选暂存区，等待管理员批量确认后再正式入库。

## 6. MVP 范围

### 6.1 包含

- 单页面问答入口。
- 基本模式与深度模式两个明确入口。
- 基础 API 调用入口，供外部 Agent 和受控客户端复用。
- 标准目录查询入口，用于查询知识库中是否已有某个标准。
- 问题输入和回答展示。
- 本地知识库检索结果接入。
- 答案引用展示。
- 检索依据摘要展示。
- 来源类型和正文可访问状态展示。
- 证据不足提示。
- 用户反馈：有用、无用、引用错误、证据不足。
- 登录用户的服务端会话历史；对“还有哪些文件”“该标准是否还有规定”等依赖上下文的追问，先结合上一轮用户问题补全检索查询，再进入领域判断和检索。
- 邀请码与邮箱验证注册、登录、用户 API Key、每日次数配额和调整记录。
- 领域相关性判断：无关问题直接拒答，不进入检索、联网、OCR 或补库记录。
- 知识库缺口记录：相关问题无证据时，可异步记录为补库候选任务。

### 6.2 不包含

- 完整用户上传与管理员审核流程，作为 V1 功能预留。
- 完整后台知识库管理界面。
- 完整组织/企业权限体系。
- 任何充值、订阅、支付、退款或发票功能。
- 复杂报告生成。
- 全自动标准入库审批。
- 多租户公共网关和复杂组织权限体系。

## 7. 功能需求

### 7.1 问答输入

用户可以在问答页面输入自然语言问题。

要求：

- 支持中文专业问题。
- 支持较长问题，至少 1000 字。
- 支持回车提交或按钮提交。
- 提交后显示加载状态。
- 防止重复提交。

问答入口必须先进行领域相关性判断。

属于服务范围的问题包括：

- 矿产资源、地质勘查、矿山设计、矿业权管理、自然资源管理相关问题。
- 标准、规范、规程、政策、技术要求、条款解释和标准状态核验。
- 与矿产资源技术报告、资源储量、工程间距、勘查类型、控制程度等业务相关的问题。

不属于服务范围的问题包括：

- 通用数学、闲聊、写作、翻译、编程、娱乐等与本产品领域无关的问题。
- 明显试图把系统当作通用大模型调用的问题。
- 恶意提示词注入、要求泄露系统提示词、绕过限制或消耗 token 的请求。

无关问题处理：

- 直接返回固定拒答，例如“本服务仅回答矿产资源标准规范及相关政策技术问题，无法处理该问题”。
- 不调用知识库、联网搜索、OCR、多模态模型或长上下文推理。
- 不记录为知识库缺口或补库任务。
- 可记录最小化的安全计数和调用日志，用于限流和风控。

#### 7.1.1 基本模式与深度模式

问答入口提供两种由用户主动选择的工作模式，默认使用基本模式，并在浏览器本地记住用户上次选择。

**基本模式 · 快速查证**

> 适合查询标准、政策文件和办事指南中的明确内容，包括条款原文、术语定义、技术数值、材料清单、办理依据和官方来源。通常快速返回，不进行全库逐份审查。

**深度模式 · 综合研究**

> 适合跨标准汇总、逐项对比、差异与冲突检查、复杂条件分析，以及需要扩大检索范围的问题。系统会在知识库可用范围内逐份检索、提取证据并生成研究结果，所需时间较长。

两种模式共同遵守：

> 两种模式均以知识库证据为依据。知识库缺少必要正文时，系统会明确说明证据不足，不使用模型常识代替标准原文。

交互要求：

- 模式选择使用输入框上方的分段控件，不隐藏在设置菜单中。
- 基本模式识别到跨文件审查、完整性核验或复杂差异比较时，可以建议切换深度模式，但不能擅自扣除深度模式配额。
- 用户可从基本模式答案一键转入深度模式，保留原问题和会话上下文。
- 深度模式创建任务前明确提示“本次将消耗 3 次，预计耗时较长”。
- 深度模式采用异步任务，页面展示当前阶段、进度、已审查文件数、候选文件总数和证据覆盖数；用户离开页面后任务仍可继续。
- 深度模式不是简单提高模型推理强度，也不是把基本模式的 `top_k` 调大；它必须使用独立的候选全集枚举、逐文件证据提取、结构化比较和结果汇总流程。

### 7.2 标准目录查询

用户可以查询知识库中是否已有目标标准。

支持查询条件：

- 标准号。
- 标准名称。
- 标准状态。
- 文档类型。
- 是否有正文。
- 是否 OCR 完成。
- 是否可用于问答。

结果展示：

- 标准名称。
- 标准号。
- 状态。
- 来源。
- 正文可访问状态。
- 是否已解析入库。
- 入库时间。
- 是否可用于问答。

该功能应进入 MVP，因为用户需要先知道库中有没有目标标准。

### 7.3 答案生成

系统根据检索证据生成结构化答案。

答案必须包含：

- 直接结论。
- 依据来源。
- 条款、章节或页码。
- 原文引用或证据摘要。
- 适用条件。
- 不确定性说明。

如果没有条款级证据，答案必须降级为说明性回答，不能生成确定性规范结论。

如果问题属于服务范围但知识库没有可用证据，系统可以返回知识库缺口响应：

- 明确说明当前知识库暂未找到可用于回答的条款级依据。
- 不等待联网搜索、OCR 或人工审核完成。
- 可生成 `knowledge_gap_task`，进入后台低优先级补库队列。
- 允许用户通过任务编号或后续再次查询获取更新结果。

知识库缺口任务不等于答案证据。任务处理完成并经管理员审核入库前，不能作为公共问答依据。

### 7.4 引用来源展示

每条引用至少包含：

- 文件名称。
- 标准号或文档编号。
- 来源类型。
- 正文可访问状态。
- 章节或条款。
- 页码。
- 引用片段。
- 来源 URL 或本地文档 ID。
- 检索得分。

### 7.5 检索依据展示

系统应展示本次回答使用的检索概况：

- 全文搜索命中数量。
- 向量检索命中数量，如已启用。
- 知识图谱命中数量，如已启用。
- 联网搜索命中数量。
- 是否存在条款级证据。
- 是否存在来源冲突。

### 7.5.1 意图理解、术语归一和证据校验

系统不能只依赖关键词检索。对于自然语言问题，应在进入检索和答案生成前做轻量意图理解和术语归一。

第一阶段至少支持以下意图类型：

- `authority_responsibility`: 权限归属、负责机构、哪一级部门办理、谁负责。
- `service_materials`: 矿业权登记、延续、变更等事项需要提交的材料或资料清单。
- `service_procedure_basis`: 矿业权事项的办理依据、办理程序和适用文件。
- `legal_responsibility`: 报告真实性、弄虚作假等法定责任主体。
- `projection_numeric_rule`: 有限/无限外推的比例、距离基准和适用条件。
- `related_documents`: 基于上一轮主题继续查找其他文件或相关规定。
- `standard_selection`: 某矿种、事项或报告应使用哪个标准或规范。
- `numeric_table_lookup`: 工程间距、规模划分、指标值、表格交叉读取。
- `clause_comparison`: 不同标准、不同矿种或不同版本的条款差异。
- `status_verification`: 标准或政策是否现行、是否废止、是否替代。
- `definition_explanation`: 术语定义、分类关系、适用条件解释。
- `out_of_scope`: 与矿产资源、标准规范、自然资源管理无关的问题。

系统应维护领域术语库 `domain_lexicon`，用于核心术语、业务等价关系、检索扩展、反例和模型不可用时的降级。领域词典不能取代大模型对错别字、口语、省略、否定、纠正和连续对话目标的理解。

权限归属问题必须同时维护三个相互独立的语义角色：

- `license_issuer_level`：当前有效勘查许可证或采矿许可证的颁发层级。
- `mining_right_granting_level`：矿业权出让、配置或登记权限层级。
- `filing_authority`：矿产资源储量评审备案负责机关；现行规则按许可证颁发层级确定。

矿种、矿山规模和矿业权出让权限只能作为背景条件，不能替代许可证颁发机关。确定性角色已经明确时直接检索和回答；角色冲突或表述含混时，允许 DeepSeek 在首次检索前补充结构化角色，但不得覆盖已确定角色或根据矿种、规模猜测发证机关。

示例：

| 用户表达 | 标准表达 | 意图 |
| --- | --- | --- |
| 采矿证 | 采矿许可证、采矿权 | `license_reference`，由问句中的“材料/办理/评审机构”等动作继续确定业务意图 |
| 探矿证 | 勘查许可证、探矿权 | `license_reference`，不能仅凭证件词触发权限路由 |
| 储量报告评审 | 矿产资源储量评审备案 | `authority_responsibility` |
| 去哪个机构、找谁办、谁负责 | 负责部门、权限归属 | `authority_responsibility` |
| 沙金 | 砂金、金属砂矿类 | `standard_selection` |
| 工程距离 | 勘查工程间距 | `numeric_table_lookup` |
| `1类型`、`I类型`、`Ⅰ类型`、`一类型` | `Ⅰ类型` | `numeric_table_lookup` |

认证、限流和低成本领域门控通过后，大模型作为第一语义阶段处理当前问题和最近最多 4 轮用户问题，负责领域错别字校准、口语归一、意图理解、否定/纠正语义和必要的歧义判断。常见 Unicode 数字等无语义机械转换可在模型前执行；确定性规则只保留不会改变业务含义的最小兜底。确认后的模糊、比较、跨文件和关系型问题再由 DeepSeek 生成结构化查询计划。计划至少包含规范化问题、意图、文档类型、检索扩展、排除概念、证据关系组和搜索模式，并由系统白名单与默认 Schema 校验。

查询理解采用分层策略：先做认证、限流和廉价领域门控，再由大模型结合当前问题与最近用户问题形成规范化问题；业务 Schema 随后保护标准号、文号、数值、事项类型和高价值关系，并决定是否需要用户确认。复杂问题在首次检索前理解目标关系，不能等到答案生成阶段才介入。第一轮证据不足时，证据审查器可给出一次 refined query，系统最多执行第二轮检索。默认不使用 HyDE 或无边界多查询生成。

采矿许可证办理材料属于受保护业务 Schema：未说明办理类型时必须按新立、延续、变更、注销四类确认，不得按自然资源部/省级发证机关分叉。发证机关只在用户明确询问矿产资源储量评审备案负责机构时作为决定条件。用户明确表示“与评审备案无关”等否定或纠正时，系统必须恢复最近仍有效的采矿证办理目标，不能沿用上一轮错误事项。

政策正文附件属于可问答知识范围，不得因 `document_type=policy_attachment` 被深度研究目录排除。采矿权申请材料问题必须纳入自然资规〔2023〕4号附件4《采矿权申请资料清单及要求》，按办理类型逐项检索材料、要求和提交形式。

“完成事项 A 后、取得结果 B 前还需办理什么”属于流程阶段关系问题。系统必须把 A 作为已完成背景、把 B 作为目标动作，围绕 B 的办事指南、申请材料目录、缴费或有偿处置要求检索，不能被 A 对应的服务指南劫持，也不能默认转成跨法规比较。对于“资源储量评审备案后、领取采矿证前还需什么手续”，基本和深度模式均应优先核对采矿权登记办事指南，并把材料名称转换为可执行动作；例如“矿业权出让收益（价款）缴纳或有偿处置证明材料”应表达为“完成缴纳或有偿处置并取得证明材料”。

第一语义阶段的大模型只负责错别字校准、规范化问题、业务意图、阶段关系、否定纠正和歧义判断，不得先自由生成最终专业答案再反向寻找证据。答案先行会造成检索锚定和模型常识污染；实际测试中，无证据自由回答遗漏了矿业权出让收益事项，并加入了未由目标办事指南支持的环评、安全预评价等内容。正确流程是“模型理解目标 -> 本地受控检索 -> 证据充分性校验 -> 基于证据生成答案”。

本地证据确实不足时，后续可增加白名单官方站点的受控全文补充：只允许自然资源部、国家标准平台和已审核来源，先抓取并结构化证据，再生成明确标注来源的回答。不得把模型训练知识标注成“互联网内容”，也不得在没有实际联网证据时输出“基于互联网回答”。当前办事指南已完整入库，因此本问题不依赖运行时联网。

确定性识别出的高价值关系意图、用户明确给出的标准号和经过验证的证据条件属于受保护约束。大模型只能补充同义词、专业表达和检索提示，不能覆盖受保护意图或删除证据条件。大模型推测的文件标题仅作为软排序提示，不得直接排除其他文件；软范围证据不足时必须保留全库混合检索能力。

检索性能采用分级预算：普通问题默认返回 10 条、跨标准比较返回 20 条；内部召回按确定性文档范围、普通全库查询、跨标准比较和完整材料清单分别设置预算，不能用统一的大 `top_k`。证据审查模型最多接收 10 条紧凑候选，并只负责选择最多 4 条代表性直接证据，不同时生成最终答案。

结构化查询规划和证据筛选应关闭模型思考模式，默认 `max_tokens` 分别为 600 和 800；最终答案默认 1000。定义和概念类问题按 7.5.4 根据直接定义原文长度和定义槽位数量动态分配 `max_tokens`，不能简单套用全局固定值。一次请求内的模型阶段复用 HTTP 连接，并只传递回答所需的紧凑检索计划。不得为了降低延迟截断必要的定义原文、材料清单、表格行列或多标准比较证据。

在线 embedding 优先通过 ANN 索引检索。ANN 元数据一致性校验可短时缓存；ANN 或 embedding 不可用时，精确 JSON 向量和本地 N-gram 哈希向量只允许在 SQL 已缩小范围且候选不超过 100 条时扫描，超限时依赖 FTS、知识图谱和异步补库，不进行大范围逐条余弦计算。

`chunk_embeddings` 使用 `(chunk_id, vector_model)` 复合主键，其中 `chunk_id` 为首列，已满足按 chunk 回表的索引需求，不应再增加重复索引。知识图谱模糊检索只针对标准、矿种、机构、材料、表格等命名实体，条款正文由 FTS/ANN 负责；KG 连接应先物化小实体候选集，再连接关系和 chunk。

高价值关系问题采用证据槽位判断充分性。例如探矿权转采矿权问题至少区分经评审备案报告、勘查程度条件和报告类型限制；伴生矿产资源量类型问题至少区分基本分析且研究充分、基本分析但研究不足、仅组合分析三种情形。证据槽位已经由条款满足时，大模型不得因原文没有重复用户口语表达而判定证据不足。

对于“矿种 + 勘查类型 + 工程间距”等目标明确的问题，应先利用知识库 schema 和标准元数据锁定一个或少量候选标准，再在候选文档内检索条款或表格；候选不存在、证据不满足意图或路由置信度不足时，必须自动退回全库检索。对于“哪些规范存在不一致”等跨标准比较问题，不得使用单标准限定，应保留全量检索。

对于高价值意图，答案生成前必须做证据可回答性校验：

- 权限归属类问题必须命中包含负责主体的证据，例如“自然资源部负责”“省级自然资源主管部门负责”。
- 办证材料类问题必须命中对应事项和申请类型的材料表/附件；只命中“按附件执行”时只能说明适用依据，不得推测逐项材料。
- 办理依据类问题必须命中办理事项对应的现行文件或官方办事指南，不能被评审备案权限文件替代。
- 报告真实性责任必须命中“真实性负责/不得弄虚作假”等直接责任条款，不能按许可证颁发层级回答。
- 外推比例类问题必须同时命中外推类型、比例和距离基准；`1/2尖推` 与 `1/4平推` 不得脱离有限/无限外推条件混用。
- 标准适用类问题必须命中标准标题、标准号或目录证据，不能只根据训练数据推测。
- 定义和概念类问题必须命中目标术语的直接定义条款；复合表达没有独立定义时，必须分别满足组成概念的定义槽位，不能用下位术语、分类关系或模型常识替代。
- 表格数值类问题必须命中表格或等价结构化证据，并按行列交叉读取。
- 用户明确要求表格时，应优先使用结构化表格 chunk；命中“见表 E.1 至表 E.5”等引用关系后，应展开同一文件中的对应表格，并以可渲染的 Markdown 表格返回。
- 条款差异类问题必须至少命中两个可比较的条款或明确说明只能列出单一来源。
- 如果命中证据与问题意图不一致，例如用油气标准回答金矿储量报告评审机构，应降级为证据不足或重新检索，不能让模型自由解释。

后续增强方向：

- 由 KB 维护可版本化的 `domain_lexicon`，问答后端只消费其归一化结果。
- 对高价值政策条款抽取 `部门 -> 负责 -> 事项`、`标准 -> 适用 -> 矿种/事项`、`文件 -> 规定 -> 权限` 等知识图谱关系。
- v1.0 已在全文、向量和图谱召回之后接入 DeepSeek 结构化证据审查，用于判断候选是否直接表达目标关系；embedding 只负责召回，不能决定证据是否可回答。
- 检索采用按需级联：简单且目标明确的问题可在 Schema + FTS/KG 已充分时跳过向量；跨标准比较和复杂关系问题使用 USEARCH ANN 补充语义召回。
- ANN 不可用时只允许在已锁定候选文档的范围内精确计算，不得恢复全库逐条 JSON 向量扫描。
- 复杂问题的回答草稿与证据审查可在同一次结构化模型调用中完成；模型输出仍须经过证据索引和系统关系约束验证。

### 7.5.2 候选排序与证据多样性（v1.0.6 已受控实施）

优先级：普通。

v1.0.6 已接入意图感知 MMR（Maximal Marginal Relevance），但不进行全局强制重排。只有符合适用意图、没有硬 Schema 范围或用户指定标准，且前 5 条至少 4 条来自同一文档时才触发；否则保持现有 RRF/规则排序。

MMR 不得作为全局统一排序规则。适用范围包括：

- 跨标准、跨制度的条款比较和代表性差异列举。
- “还有哪些文件规定了相关内容”等扩展检索。
- 普通综合政策查询和需要多个独立来源支撑的回答。
- 正确证据已经进入候选集，但被同一文档或高度相似 chunk 挤出前列的情况。

以下意图默认禁用 MMR，或等价使用 `lambda=1.0` 保留纯相关度排序：

- 指定标准、指定条款和标准目录精确查询。
- 工程间距、工业指标等结构化表格行列读取。
- 附件材料清单、办事指南材料目录等需要同一文档多个 chunk 的完整性任务。
- 定义和概念类问题，尤其是需要保留同一标准中多个术语定义条款的任务。
- 高价值证据槽位已经明确，需要组合同一文件多个条款才能完成回答的任务。

计划排序位置为：Schema/FTS/KG/ANN 混合召回 -> 硬范围和证据槽位校验 -> 保留强制证据 -> 对剩余候选执行 MMR -> 证据审查/答案生成。MMR 只能重排已经召回的候选，不能替代问题转写、Schema 路由、召回扩展或知识库补齐。

参数策略：

- 符合条件但暂无历史数据时，从 `lambda=0.7` 开始。
- 比较类问题或前 5 条中至少 4 条来自同一文档/高度相似内容时，可降低到 `0.55～0.65`。
- 候选已经较分散、需要优先保持相关度时，可提高到 `0.8～0.9`。
- “结果相关度不够”只有在正确证据已经进入候选池时才可通过提高 lambda 改善排序；如果正确证据根本未召回，应回到查询理解和召回层处理。

持续评估门槛：

- v1.0.6 已建立 25 题基准并保留候选、预期来源和检索策略结果；后续继续加入真实用户反馈。
- 统计前 5 条候选的同文档比例、语义重复率、标准覆盖数和正确证据是否被挤出。
- 当前 25 题基准没有真实问题触发重复阈值，因此 MMR 保持按请求动态触发，不扩大到全局查询。

实现约束与验收标准：

- 仅对每次检索已经形成的 30～80 条候选计算多样性，不得触发全库向量扫描或新增远程 embedding 调用。
- 强制证据槽位、用户明确指定的标准和确定性 Schema 结果不得被 MMR 删除。
- 使用功能开关和按意图配置，支持快速关闭和对照测试。
- 比较/综合查询的代表性标准覆盖率提高，前 5 条语义重复率下降。
- 精确条款、表格和材料清单回归结果不得下降。
- MMR 层新增 P95 延迟目标不超过 100 ms。

### 7.5.3 查询改写、多查询扩展与模型采样策略

优先级：普通。

#### 查询改写（已采用，继续增强）

受控 Query Rewriting 适合 geowiki，应继续作为查询理解层的基础能力，但不能把所有问题都交给大模型自由改写。

- 简单、目标明确和已有稳定 Schema 的问题继续使用本地机械归一、领域词典和确定性意图，不调用查询规划模型。
- 模糊、口语化、跨文件、比较和关系型问题允许由 DeepSeek 输出一条结构化 `canonical_query`，同时给出主题、必需术语、同义术语、排除术语、文档类型和证据关系组。
- 改写必须保留用户明确提供的标准号、文号、条款号、矿种、数值、比例、勘查阶段、许可证颁发主体和适用条件。
- 查询规划模型只能生成检索计划，不能把模型记忆中的答案或推测文件当作证据。
- 第一轮证据不足时，证据审查器最多生成一条 `refined_query`，系统最多执行第二轮受控检索。
- 检索轨迹应同时保存原问题、规范化问题、最终检索问题、扩展术语、是否触发第二轮以及第二轮新增的直接证据，供后续评估改写是否真正提高召回。

查询改写的验收重点不是“文本更专业”，而是正确证据召回率提高，同时不得改变用户的真实问题或扩大到无关标准。

#### Multi-Query（v1.0.6 已受控实施）

Multi-Query 不全局启用，仅作为复杂问题首轮证据不足后的受控召回增强手段，且优先级低于单查询改写和证据缺口分析。v1.0.6 最多并发执行 2 条补充查询，其中优先保留证据审查器生成的一条 `refined_query`，剩余额度用于绑定独立证据目标的模型子查询。

允许考虑启用的场景：

- 跨标准比较，需要分别检索各比较维度或各类距离基准。
- “还有哪些文件规定了相关内容”等扩展查询。
- 一个问题同时包含政策条件、技术标准限制和办理材料等多个独立证据槽位。
- 单条规范化查询已执行，但第一轮未覆盖全部必需证据组。

默认禁用的场景：

- 标准号、文号、条款号明确的精确查询。
- 工程间距、工业指标和分类表等结构化表格读取。
- 附件材料清单和办事指南完整材料目录。
- 当前 Schema/FTS/KG 已经找到充分直接证据的简单问题。

计划实现约束：

- 每次最多生成 2～3 条具有不同证据目标的子查询，不能只生成同义改写后重复搜索。
- 每条子查询必须绑定明确的证据槽位或比较维度，例如“行政许可条件”“报告类型限制”“外推距离基准”。
- 用户明确的标准号、矿种、时间范围和受保护意图必须继承到每条子查询。
- 子查询结果使用现有 RRF、规则评分和证据校验合并，并按 `chunk_id`、文档和语义重复去重。
- 第一条查询已满足全部证据槽位时立即停止，不执行额外子查询。
- Multi-Query 只负责提高召回，不直接生成答案；不得引入 HyDE 文本作为证据。
- 结构化子查询生成使用 `temperature=0`、关闭思考模式并限制输出长度。
- 实施前必须通过离线评估证明新增查询带来了新的直接证据，而不是只增加重复候选、模型调用和延迟。

#### Temperature（采用当前低随机性策略）

当前温度策略适合专业标准问答，应在 PRD 中固定为默认基线：

- 查询规划、查询改写、Multi-Query 生成和证据审查等结构化 JSON 调用使用 `temperature=0`。
- 最终自然语言回答默认使用 `temperature=0.2`，允许配置在 `0～0.3` 范围内，不应为了语言多样性提高到高随机区间。
- `definition_explanation` 使用确定性模板时不调用最终回答模型；确需模型组织语言时固定使用 `temperature=0`，不得继承普通回答的 `0.2`。
- 温度参数后续可以环境变量化，但生产默认值和允许范围必须由后端控制，不能由普通 API 用户任意指定。
- 温度调整只影响生成稳定性，不得作为解决检索缺失、证据错误或来源重复的手段。

#### USEARCH expansion_search（保留当前基线，离线调优）

v1.0.6 ANN 基线为 `connectivity=24`、`expansion_add=128`、运行时 `expansion_search=64`。其中 `expansion_search` 是 USEARCH/HNSW 的图搜索广度参数，不属于查询改写、多查询扩展或 MMR。

- 25 题、22,778 向量离线基准比较了 `64/96/128`；三者最低 Recall@20 均为 `0.95`，平均 Recall@20 分别为 `0.994/0.996/0.996`，P95 分别为 `0.518/0.605/0.646 ms`。
- 当前选择 `64` 作为运行时及后续建索引基线；参数仍可通过环境变量和离线工具重新评估。
- 如果提高搜索广度没有带来可测量的正确证据提升，应保留较低成本参数。
- ANN 参数调整必须与索引版本、embedding 模型和知识库快照一同记录并重新发布。

### 7.5.4 定义和概念类问答（v2.0.0 已实施）

优先级：普通。

v2.0.0 已实现受保护意图 `definition_explanation`、定义槽位、复合术语拆分、同一标准互补条款保留和确定性原文模板。定义证据完整时跳过最终回答模型；确需模型兜底时固定使用 `temperature=0` 和动态输出预算。以“资源储量的定义”为验收问题，系统只选择 GB/T 17766-2020 第 2.7 条“资源量”和第 2.12 条“储量”，排除第 2.14 条“证实储量”和关系图。

#### 意图识别与目标术语

系统应确定性识别以下常见问法：

- `X 的定义`、`X 如何定义`、`X 是什么`、`什么是 X`。
- `X 是什么意思`、`X 的概念`、`X 术语定义`。
- 询问多个概念的区别、分类关系或上下位关系，但核心任务仍是核验标准定义。

查询计划至少增加：

- `target_terms`：用户直接询问的一个或多个目标术语。
- `definition_mode`：`exact`、`compound`、`classification` 或 `comparison`。
- `definition_slots`：每个必须召回的定义术语及其证据状态。
- `preferred_definition_sources`：由 Schema、标准目录或术语索引确定的优先标准，不允许仅由模型猜测。

对于“资源储量”等行业常用复合表达，系统应先检查现行标准是否存在同名独立定义。不存在时，才按受治理术语关系拆分为“资源量”和“储量”等组成概念，并分别建立定义证据槽位；不能直接把“证实储量”等下位术语当成“储量”的定义。

#### 定义检索与知识库 Schema

定义类问题优先采用 Schema + 精确条款/标题检索，不默认调用 ANN、查询规划模型或无边界扩展。推荐由 KB 建立可维护的 `term_definitions` 结构，至少包含：

- `term`、`canonical_term`、`aliases`、`term_type`。
- `definition_text`、`document_id`、`standard_no`、`clause_no`。
- `document_status`、`source_url`、`validation_status`、`updated_at`。
- 复合术语与组成术语、上位术语、下位术语之间的受治理关系。

如果暂未建立独立术语表，也必须利用现有 clause-level chunk 对“条款号 + 术语标题 + 定义正文”做精确匹配。定义证据槽位未满足时，可执行一次受控 refined query；Multi-Query 仅允许分别补齐不同定义槽位，不能生成多个同义问法重复检索。

定义类问题必须禁用 MMR，或等价使用 `lambda=1.0`。同一标准中的多个定义条款属于完整答案所需的互补证据，不能因文档相同或语义相近而去重删除。

#### 证据和回答约束

每个定义槽位必须包含：标准名称、标准号、具体条款、定义原文、标准状态和官方来源。答案按以下顺序组织：

1. 说明用户术语是否存在同名、独立的标准定义。
2. 逐项引用目标术语或组成术语的直接定义原文。
3. 在原文之后单独给出简短解释、区别或关系说明。
4. 关系条款、分类图表和行业习惯只能作为补充，不能替代定义原文。

定义引文只允许清理多余空格、断行和已确认的 OCR 错误，不得改写、缩写或加入原文没有的限定条件。解释部分必须与原文明确分隔；模型训练数据、行业印象和推断不得写成标准定义。

当定义槽位完整、回答结构固定时，应优先采用确定性模板生成答案并跳过最终回答模型，以降低幻觉、延迟和 token 成本。

#### 温度与动态输出预算

定义类确定性模板不使用生成温度。确需大模型组织语言时，使用 `temperature=0`。

模型兜底回答的 `max_tokens` 按直接定义长度和定义槽位数量动态计算：

```text
definition_max_tokens = round_up_100(
  clamp(
    650 + 1.5 * definition_quote_chars + 100 * definition_count,
    1000,
    1600
  )
)
```

其中 `definition_quote_chars` 为最终选中定义原文清理后的总字符数，`definition_count` 为必须输出的定义槽位数量。单一定义通常使用 1000；两个至三个定义通常使用 1100～1400；较长分类体系最高不超过 1600。官方来源链接由后端在生成后追加，不占用回答模型预算。

以“资源储量的定义”为例，GB/T 17766-2020 第 2.7 条“资源量”和第 2.12 条“储量”直接定义合计约 216 字，两个定义槽位按公式使用约 1200 `max_tokens`。提高预算只能防止完整定义和固定格式被截断，不能替代正确意图和证据召回。

#### 生成诊断与截断处理

每次模型调用应记录：

- `finish_reason`、`prompt_tokens`、`completion_tokens`。
- 实际 `temperature`、实际 `max_tokens` 和 Prompt 版本。
- 是否因 `finish_reason=length` 被截断。

定义证据完整但模型因长度截断时，允许在 1600 上限内执行一次受控重试；重试只能增加输出预算，不能更换已经验证的定义证据。

#### 验收标准

- “资源储量的定义”必须明确说明是否存在独立定义，并同时引用 GB/T 17766-2020 第 2.7 条“资源量”和第 2.12 条“储量”，不得用第 2.14 条“证实储量”替代。
- “资源量的定义”“储量的定义”“证实储量的定义”分别命中对应直接定义条款。
- 同一标准中的多个定义条款不会被 MMR 或文档级去重删除。
- 原文引文与知识库直接定义一致，解释不得加入原文没有的事实条件。
- 定义槽位完整时优先走确定性模板，并跳过不必要的 ANN、Planner 和最终回答模型。
- 回归测试覆盖同义问法、复合术语、上下位术语、多个定义比较和定义缺失场景。
- trace 可以明确判断回答是否发生 token 截断。

### 7.5.5 深度模式跨文档研究工作流（v2.0.0 已实施）

优先级：普通。

v2.0.0 已完成持久化研究任务、候选目录枚举、逐文件限定检索、AND 证据组校验、结构化事实抽取、比较矩阵、覆盖说明、重启恢复和配额事务。结构化事实默认每批处理 4 条证据；模型 JSON 截断或格式错误时自动拆分重试，最终失败时仍保留直接条款为可验证事实，不得误标为“证据不足”。对于“无限外推”“有限外推”等受保护关系，用户限定范围会在模型规划后再次由代码强制校验，摘要和事实均不得串用相反类型，也不得根据单个片段未出现某内容就断言整份文件“未规定”。v2.0.1 删除对比表后的重复原文列表，完整引文继续通过结构化 `sources` 和前端折叠来源提供；同时默认排除第 2 章“规范性引用文件”等标准目录片段，除非用户明确询问引用关系。

深度模式面向跨标准汇总、差异与冲突检查、复杂条件分析和基本模式无法完成的完整性任务。后端新增受保护研究意图 `cross_document_audit`，并采用持久化异步任务，不占用普通 HTTP 请求等待整个研究过程。

公开任务接口：

```http
POST /api/research/tasks
GET  /api/research/tasks/{task_id}
GET  /api/research/tasks/{task_id}/result
POST /api/research/tasks/{task_id}/cancel
```

研究工作流至少包括：

1. 识别用户要求核验的基准文件、主题、比较范围和比较维度。
2. 从知识库 Schema/标准目录枚举完整候选文档集合，而不是只把普通检索前若干条结果当作全集。
3. 从基准条款中提取比较维度；没有明确基准文件时，先形成受证据约束的通用比较维度。
4. 在每个候选文档内部执行限定范围检索，记录已审查、未命中和正文不足状态。
5. 将直接条款转换为结构化事实，至少保留文件、条款、适用范围、条件、数值/要求、短引文和官方来源。
6. 按比较维度分类为：一致、更严格、更宽松、表述不同但实质等价、适用范围不同、特别规定、未规定、证据不足、疑似冲突需人工复核。
7. 输出结论摘要、对比矩阵、代表性短引文、官方链接、已审查/候选文件数量和知识库快照。

实现约束：

- 候选全集、检索结果、任务状态和最终结果均不得暴露知识库文件、chunk 或向量库的批量下载能力。
- 逐文件检索可以并发，但云端默认限制为单用户最多一个运行中的深度任务，并设置全局工作并发上限，避免低配服务器被批量任务占满。
- 任务状态持久化到应用数据库；服务重启后，未完成任务应回到可恢复队列，不能静默丢失。
- 研究结果必须说明覆盖边界。候选文档被上限截断、正文不可用或部分检索失败时，状态应为 `partial` 或 `insufficient_evidence`，不能伪装成完整审查。
- 深度模式继续复用账号、API Key、会话、反馈、标准来源、FTS、KG、ANN 和证据治理能力，但拥有独立的任务接口和编排层。
- 深度模式的结构化规划和事实抽取使用 `temperature=0`；最终摘要保持低随机性，并且只能使用已提取事实和引用。

深度任务状态至少包括：

- `queued`：已入队，尚未开始。
- `planning`：正在识别基准、范围和维度。
- `retrieving`：正在枚举和逐文件检索。
- `analyzing`：正在提取事实和比较。
- `completed`：证据覆盖满足任务要求并已生成结果。
- `partial`：已生成可用结果，但覆盖不完整。
- `insufficient_evidence`：候选范围存在，但缺少足够正文证据。
- `failed`：系统错误，任务未形成可用结果。
- `cancelled`：排队阶段由用户取消。

验收标准：

- 用户可以在网页和 API 中创建、轮询、读取和取消深度任务。
- 任务结果包含比较矩阵、直接短引文、官方来源、覆盖统计和知识库快照。
- 相同用户不能同时创建多个运行中的深度任务。
- 任务重启恢复、失败退款、排队取消退款和结果访问权限均有自动化测试。
- 基本模式答案建议转深度模式时，一键转换保留问题和会话，并只补扣 2 次，使该问题总消耗为 3 次。

### 7.5.6 分阶段 Prompt 校准层（低优先级）

优先级：低。

Prompt 校准层用于把已经验证的领域理解规则、正反例和输出约束按检索阶段与意图动态注入模型，以较低成本改善第三方大模型在固定业务模式下的稳定性。它只能模拟部分 LoRA 行为，不修改模型权重，也不能替代知识库、Schema、召回、Chunk 治理和证据校验。

适合校准的阶段：

- 查询规划：区分背景条件与真正判断依据，例如矿山规模与储量评审权限、详查程度与报告名称。
- 证据审查：区分直接证据和只包含相同关键词的旁证，例如父政策附件指引与附件正文、普通工程间距表与外推距离规则。
- 最终回答：统一短引用、适用条件、来源展示、证据不足表述和比较类输出格式。

不适合交给 Prompt 校准解决的问题：

- 知识库未收录目标文件或文件状态错误。
- Chunk、表格、页码和附件关系解析错误。
- Schema 范围、FTS/KG/ANN 召回不足或正确证据根本未进入候选集。
- 数据库、embedding、ANN 和模型网络调用的性能问题。

计划实现方式：

- 建立版本化 Prompt Registry，按阶段和意图保存配置，例如 `planner/authority/v1`、`reranker/projection/v1`、`answer/comparison/v1`。
- 每次只注入当前意图需要的短规则以及 1～3 个经过人工确认的正例/反例，禁止形成不断膨胀的全局 Prompt。
- 受保护意图、标准号、文号、数值、证据槽位和系统安全规则继续由代码校验，Prompt 无权覆盖。
- 查询规划和证据审查继续使用 `temperature=0`、关闭思考模式和结构化输出。
- 所有 Prompt 版本必须支持功能开关、灰度测试、回滚和检索轨迹追踪。
- Prompt 中不得嵌入未获授权的标准全文；示例优先使用抽象化、短引文或公开许可内容。

实施门槛与验收：

- 至少积累 30～50 个已经明确属于“模型看到正确证据但理解、筛选或表达错误”的失败案例。
- 按查询规划、证据审查和答案表达分别建立校准集，不能把检索缺失案例混入 Prompt 效果评估。
- 与现有 Prompt 做离线和灰度对照，统计意图准确率、直接证据选择准确率、引用错误率、答案稳定性、token 和延迟变化。
- 只有在目标错误类型上有稳定提升，且不降低现有高价值回归集准确率时才上线。
- Prompt 校准属于低优先级增强，不得早于知识库缺口、系统性召回错误、材料/表格完整性和普通优先级排序优化。

### 7.5.7 大模型主导的问题歧义确认（普通优先级）

用户表达的变化无法依靠领域词典穷举。系统将查询理解调整为“大模型主判语言歧义、业务 Schema 和受保护规则负责校验、领域词典负责核心术语与降级”的混合模式。

v2.2.0 状态：已完成。基本模式、深度模式、网页和 API 已共用同一歧义确认契约；确认阶段不执行知识库检索、不创建补库或研究任务、不扣问答次数。

处理流程：领域门控 -> 会话上下文补全 -> 机械归一 -> 大模型结构化歧义判断 -> 后端业务规则校验 -> 确认或进入检索。大模型只能理解问题、识别缺失条件和生成候选解释，不得回答专业问题或使用训练数据替代知识库证据。

- 只有不同解释会改变目标标准、条款范围、业务事项或结论时才触发确认；明确问题不得增加无意义交互。
- 高置信、字段完整且已有稳定 Schema 的问题继续走快速路径，避免为工程间距表格等明确查询增加模型延迟。
- 候选解释为 2～4 个互斥的完整检索问题，同时允许用户自由补充。
- 意图确认不扣问答次数，不执行 Dense Embedding、证据审查、深度任务或补库；深度模式在确认后才预留 3 次额度。
- 基本模式、深度模式、网页和 API 共用同一问题解析结果与结构化契约。
- 模型不可覆盖行业等价关系、权限角色、标准号、证据槽位和安全规则；模型不可用时退回确定性理解和领域词典。对于已有稳定业务 Schema 的高价值问题，模型超时或结构化输出失败也不能跳过必要确认。
- 记录模型理解、候选方向、用户最终选择和后续检索结果，用于评估过度确认率、漏确认率、意图准确率、额外 token 和延迟。
- 后端对模型结果执行受保护约束校验：用户明确给出的标准号、标准名称、勘查类型和许可证颁发层级不得被删除或改写；无效、领域外、重复或未补齐关键条件的候选不展示。
- 对“储量评审备案机构”这类已有稳定业务 Schema 的高价值问题，许可证颁发机关缺失时由 Schema 兜底触发确认；候选只描述待核验情形，不预先输出结论。
- 对“采空区怎么处理”这类只给主题和宽泛动作、未说明具体目标的问题，即使模型漏判为明确问题，后端 Schema 仍按稳定性、积水水害、塌陷监测和工程治理给出候选；已明确具体目标时不得重复确认。
- 明确的工程间距、定义、材料条款和已给出许可证颁发机关的权限问题继续走快速路径，不增加歧义模型延迟。

### 7.5.8 受控工具编排与 `tool_call` 决策

当前系统不开放由大模型自由选择任意网络、文件或系统工具的通用 `tool_call`。现有 Agent 已具备受控编排能力：问题理解、业务 Schema 校验、知识库 Schema/FTS/KG/ANN 检索、证据审查、补充检索、深度研究任务和补库记录均由服务端按确定性权限与成本边界调度。是否属于合格的垂直智能体，取决于能否完成“理解 -> 规划 -> 调用受控能力 -> 审查证据 -> 输出或追问”的闭环，而不取决于是否暴露某个供应商的 `tool_calls` 字段。

- 大模型负责语言理解、结构化计划和受证据约束的归纳；服务端负责决定实际可调用能力、参数校验、超时、配额、权限和审计。
- 私有知识库、用户次数和后台补库任务不得由模型绕过 API 直接访问或修改。
- 当前阶段继续使用显式工作流，避免自由工具调用造成越权、不可预测成本、重复检索和难以复现的答案路径。
- 后续需要扩展时，可以增加白名单工具注册表，例如标准目录查询、私有知识库检索、官方来源查询和深度研究任务；每个工具必须声明输入 Schema、调用权限、超时、最大次数、配额成本、日志字段和失败降级。
- 不向模型提供 Shell、任意文件读取、任意 URL 抓取、知识库写入或管理员操作等开放能力。

### 7.5.9 统一问题分类与检索策略注册表（待修复）

优先级：普通。

状态：待实施。v2.2.3 反馈审计确认，当前系统虽然已有问题理解、领域词典、基本模式检索规划和深度研究规划，但分类结果分散在多个组件中，尚未形成基本模式与深度模式共同消费的唯一业务契约。该问题会造成知识库已有直接证据却选错文档、深度模式重新解释问题、领域词典跨意图污染查询，以及知识缺口与召回失败无法区分。

本次修复不继续增加针对单个问法的零散正则。系统应建立统一的 `QueryClassification`，在任何知识库检索或深度研究目录枚举之前只生成一次，并在整次请求中保持稳定。基本模式和深度模式必须共享该结果；深度模式只能扩大候选文件范围、增加逐文件审查和比较深度，不能改变 `primary_intent`、目标事项或用户明确限定的对象。

统一分类至少包含：

- `primary_intent`：主要问题类型。
- `secondary_intents`：同一问题中需要补充处理的次要类型。
- `target_entity`：目标标准、政策、矿业权事项、术语、表格或技术对象。
- `mineral`：矿种或矿种类别。
- `business_action`：新立、延续、变更、注销、转采、评审备案、压覆审批等动作。
- `completed_stage` 与 `target_outcome`：已经完成的环节和用户希望达到的结果。
- `document_types`：应优先检索的标准、政策、附件或办事指南类型。
- `evidence_slots`：回答前必须满足的证据槽位。
- `output_shape`：目录、定义、数值、材料清单、流程列表、权限结论或比较矩阵。
- `ambiguities` 与 `missing_slots`：确实会改变结论、需要用户确认的缺失条件。
- `confidence`：分类置信度和降级原因。

首批主要问题类型和检索策略如下：

| 主要类型 | 典型问题 | 首选检索策略 |
| --- | --- | --- |
| `standard_catalog_lookup` | 某矿种适用哪个标准、标准是否现行 | 标准目录和元数据精确匹配优先，正文检索作为补充 |
| `clause_lookup` | 某标准或条款规定了什么 | 用户指定标准内的条款号、标题和 FTS 精确检索优先 |
| `definition_lookup` | 某术语的定义是什么 | “术语和定义”章节、首选基础标准和直接定义条款优先 |
| `numeric_table_lookup` | 工程间距、工业指标、规模数值 | 先按矿种和事项锁定标准，再完整读取表头、行、列和单位 |
| `service_materials` | 新立、延续、变更需要哪些材料 | 办事指南和政策附件优先，按办理类型返回完整材料目录 |
| `service_workflow` | 办理流程、下一步手续、如何完成事项 | 依次检索适用范围、申请条件、材料、流程、时限和办理结果 |
| `authority_jurisdiction` | 哪一级负责、去哪里申请 | 按具体权限关系加载独立 Schema，区分发证、出让、备案和压覆审批 |
| `eligibility_condition` | 是否可以转采、是否满足申报条件 | 检索一般政策、适用条件、例外、特别规定和限制性条款 |
| `cross_document_comparison` | 不同标准有哪些差异或不一致 | 先从目录枚举文件，再逐文件限定检索并核验覆盖率 |
| `technical_method` | 采空区圈定、估算方法选择等技术问题 | 先识别具体目标、阶段和约束；确有歧义时确认后再检索技术规范 |
| `out_of_scope` | 与地质矿产领域无关的问题 | 不检索知识库，不消耗正式问答次数 |

统一处理顺序应为：认证和限流 -> 廉价领域门控 -> 标准号、文号、数值等机械提取 -> 大模型结构化理解、错别字纠正和问题分类 -> 业务 Schema 校验 -> 必要的歧义确认 -> 按分类加载检索策略 -> 领域词典执行分类内的术语扩展 -> 候选文档锁定 -> FTS/KG/ANN 受控召回 -> 证据槽位校验 -> 回答生成。

领域词典只能在分类完成后执行，并且每个词条的扩展、负面词和证据要求必须限定在适用的 `primary_intent` 或上下文中。背景词条不得把评审备案、权限归属等证据词注入基本分析、数值表格或其他无关查询。

检索策略注册表应由配置或显式代码结构统一维护，不能继续散落在问题转写、知识库排序、证据选择和深度研究规划器的多个 `if/elif` 分支中。每个策略至少声明：允许的文档类型、候选范围来源、检索通道顺序、召回预算、必需证据槽位、是否允许 MMR/Multi-Query、输出结构和证据不足时的降级行为。

知识缺口与检索失败必须分开记录：

- 目标文件或条款确实不在知识库中，记录为 `knowledge_missing` 并进入 KB 补充任务。
- 文件和直接条款已经存在但未进入候选集，记录为 `routing_failure`。
- 条款已召回但被排序或证据审查删除，记录为 `ranking_or_validation_failure`。
- 证据正确但答案归纳错误，记录为 `synthesis_failure`。
- 分类或目标事项错误，记录为 `classification_failure`。

#### 分层业务槽位与确认状态机（待随 M16 实施）

v2.2.3 实测发现，当前确认流程只能处理单层候选。用户原始问题已经明确“采矿权扩大范围”，但第一语义模型发生结构化输出 `ValidationError` 后，降级规则未能识别“扩大范围”属于采矿权变更子类型，错误退回新立、延续、变更、注销四个一级候选。用户选择“变更”后，前端仅把候选文字作为一条新问题重新提交，没有携带原确认请求、已确定槽位或候选 ID；后端因此把一级办理类型视为完整条件，直接启动深度研究。研究只取得“变更包含5个子类型、合计50项”的汇总证据，未取得任何具体子类型的完整材料行，却仍返回 `completed/high confidence`，并在普通答案正文中再次要求用户确认，导致界面无法展示候选且错误消耗深度模式额度。

该问题必须通过通用的分层槽位和确认状态机解决，不得为“扩大范围”这一问法增加单独答案模板。

`service_materials` 分类至少维护以下业务槽位：

- `application_type`：`new`、`renewal`、`change`、`cancellation`。
- `change_subtype`：`expand_area`、`shrink_area`、`mineral_or_mining_method`、`holder_name`、`transfer`。
- `subject_scope`：油气、非油气、外商等只有在材料适用性确有差异时才要求补充的条件。
- `slot_status`：`complete` 或 `clarification_required`，并记录仍缺失的必需槽位。

大模型负责把“扩界、扩大范围、增加开采区域”等自然表达理解为业务槽位候选；业务 Schema 负责枚举校验和保护用户明确条件。即使大模型结构化输出失败，确定性降级也必须从原问题保留已经明确的业务动作和子类型，不能把具体问题降级成更宽泛的问题。

检索前必须执行按分类配置的槽位完整性校验：

- 新立、延续和注销在一级办理类型明确后可以进入材料检索。
- 变更必须同时明确 `change_subtype`，否则返回五个结构化变更候选，不得创建基本问答检索或深度研究任务。
- 原始问题已经明确“扩大范围”等子类型时，应同时填充 `application_type=change` 和对应 `change_subtype`，不再重复确认。
- 多级确认允许先确认新立/延续/变更/注销，再确认具体变更类型；所有确认轮次均不消耗问答次数。

确认协议必须传递结构化状态，而不是只重新提交候选文字：

- 响应包含 `clarification_id`、`parent_request_id`、`pending_slot`、`resolved_slots` 和候选 `slot_updates`。
- 网页和 API 客户端提交 `clarification_id + option_id` 或等价的显式槽位更新。
- 后端把选择结果合并到原始 `QueryClassification`，保留原问题中的矿种、事项、否定、标准号和其他已确认条件。
- 每次合并后重新执行槽位完整性校验，只有 `slot_status=complete` 才允许进入知识库检索和配额预留。
- 为兼容旧客户端，候选仍可保留可读的完整问题文本，但该文本不能作为状态传递的唯一依据。

材料检索必须根据已确认槽位形成硬范围。例如 `application_type=change`、`change_subtype=expand_area` 应限定到 `附件4 > 变更 > 扩大矿区范围` 下的结构化表格和材料行，完整读取材料名称、▲/—标记、特殊要求和提交形式；不得用变更类别汇总 chunk 代替具体材料清单。

证据完整性和任务状态必须联动：

- 只有汇总说明、没有目标子类型材料行时，视为 `missing_required_slot` 或 `incomplete_evidence`，不能返回高置信完成结果。
- 答案生成层不得自行在正文中追加“请确认”“还需说明具体事项”等交互指令。
- 最终文本仍包含未解决确认语义时，响应必须转换为结构化 `clarification_required`，不扣次数、不保存为已完成答案。
- 深度研究任务只有在槽位完整后才创建；确认不完整、规划失败或证据完整性门禁未通过时，已预留额度必须自动退回。
- 基本模式和深度模式共享相同的 `QueryClassification`、确认状态和必需槽位；深度模式不得重新解释或丢弃已确认条件。

诊断日志至少记录第一语义模型的结构化校验失败字段、采用的降级路径、确认层级、已解决/待解决槽位、最终策略 ID、硬文档范围和证据完整性结果，便于区分模型输出错误、状态丢失、路由失败和知识缺口。

该子任务在 `docs/QUESTION_CLASSIFICATION_REVIEW.md` 审阅完成后与统一分类架构一并实施。当前知识库已经包含采矿权变更五个子类型和扩大矿区范围的具体材料行，本案例不需要 KB 专员补库。

专项验收场景：

- “采矿权扩大范围需要什么材料”直接识别为变更-扩大范围并返回完整对应材料，不触发一级确认。
- “采矿权变更需要什么材料”返回扩大范围、缩小范围、矿种或开采方式、采矿权人名称、转让五个结构化候选，不创建研究任务、不扣次数。
- “采矿证办理需要什么材料”先确认四个一级类型；选择变更后继续确认五个子类型；选择具体子类型后只执行一次正式检索。
- 网页与 API 选择候选后保留原问题和已确认槽位，刷新或轮询深度任务时状态不丢失。
- 第一语义模型返回无效 JSON 或校验失败时，显式出现的办理类型和变更子类型仍能由受控降级保留。
- 任何 `completed` 或 `answered` 响应不得包含尚待用户确认的交互要求。

v2.2.3 已确认的基线案例包括：采矿权扩大范围办事指南已经入库但深度模式错误路由到压覆条款；GB/T 19492-2020 分类正文已经入库但未召回第 4 章直接条款；DZ/T 0205-2020 已入库但金矿基本分析查询的候选标准路由不稳定；DZ/T 0399 未入库，属于真实知识缺口。

问题分类的边界、常见表达、关键槽位、追问条件和检索策略维护在 `docs/QUESTION_CLASSIFICATION_REVIEW.md`，由用户从业务角度审核。具体反馈问题只用于验证分类器和检索策略能否泛化，不得根据单个问题预设最终答案、固定标准号或增加仅适用于该问法的分支。人工确认后的具体案例可以转为版本化回归数据，但回归样本不是运行时分类规则的来源。

验收标准：

- 基本模式与深度模式对同一问题返回相同的 `primary_intent`、目标事项和硬候选约束。
- 深度模式不得把材料、流程或权限问题默认改成跨文档比较。
- 每次请求的检索轨迹记录唯一分类结果、策略 ID、候选文档来源和失败层级。
- 对知识库已有直接证据的反馈题，正确目标文档进入候选集和最终证据集；不得以“证据不足”代替路由修复。
- 对真实知识缺口，系统明确指出缺少的目标文件或条款，不引用无关标准补位。
- 所有已确认的不满意反馈转为自动回归题，至少标注问题分类、预期来源、必须满足的答案要点和禁止出现的错误来源。
- 后台反馈只有在指定版本复测通过后才能标记 `resolved`；必须保存 `resolution_note`、`verified_version`、`verified_request_id` 和复测结果。

### 7.6 联网补充

联网能力只作为本地知识库不足时的补充。

默认交互策略是“快速返回、后台补库”。用户请求不等待完整联网搜索、OCR 或人工审核。系统可以先返回证据不足，并将真实业务问题记录为知识库缺口任务，后续由后台低优先级队列慢慢处理。

系统可以联网查询：

- 全国标准信息公共服务平台。
- 国家标准全文公开系统。
- 自然资源标准化信息服务平台。
- 其他后续确认的官方来源。

联网补充可用于：

- 标准号、标准名称、状态、发布日期、实施日期。
- 主管部门、归口单位、备案信息。
- 是否存在官方全文阅读入口。
- 候选标准发现。

联网补充不能直接用于：

- 在没有正文文本的情况下回答条款级问题。
- 将第三方文档当作官方正文。
- 忽略不同来源之间的状态冲突。

### 7.6.1 异步补库任务

当问题通过领域相关性判断，但正式知识库没有条款级证据时，系统可以创建异步补库任务。

任务用途：

- 收集真实用户需求。
- 统计高频缺失标准和高频问题。
- 后台排队执行官方来源搜索、OCR、解析和候选暂存。
- 管理员批量审核后统一入库，供后续复用。

任务不要求并发执行。MVP 阶段可以采用单 worker 或定时脚本，例如每小时处理少量任务，以降低服务器压力。

任务建议字段：

- task_id
- question
- detected_domain
- detected_keywords
- candidate_standard_no
- candidate_title
- status
- priority
- created_at
- processed_at
- result_candidate_id

任务状态建议：

- `queued`: 已记录，等待处理。
- `searching`: 正在搜索官方来源。
- `ocr_pending`: 已找到图片型或视觉型来源，等待 OCR。
- `ocr_running`: OCR 处理中。
- `review_pending`: OCR 或解析结果等待管理员审核。
- `approved_for_kb`: 审核通过，准备或已经正式入库。
- `rejected`: 不适合入库或来源不可用。
- `done`: 任务处理结束。

无关问题不能创建补库任务。

### 7.7 来源可访问性分类

系统必须为每个来源标记 `source_type`：

- `official_metadata`: 官方元数据。
- `official_fulltext`: 官方正文可文本抽取。
- `official_visual`: 官方正文存在，但为图片、切片或 canvas/PDF.js 渲染。
- `third_party_candidate`: 第三方候选文档。
- `local_kb`: 本地知识库文档。
- `unavailable`: 未找到可用来源。

系统必须为每个来源标记 `text_access`：

- `metadata_only`
- `html_text`
- `pdf_text`
- `image_ocr_required`
- `ocr_text`
- `unavailable`

### 7.8 来源冲突提示

当不同来源对标准状态、发布日期、实施日期、废止状态或替代关系给出不同信息时，系统必须提示冲突。

示例：

```text
该标准状态存在来源差异：SAMR 行业标准页面显示废止；自然资源标准化信息服务平台详情页显示现行标签，但时间轴包含废止节点且未给出废止日期。建议以主管部门最新公告或正式出版信息进一步核验。
```

状态优先级：

- 国家标准优先采用国家标准全文公开系统和全国标准信息公共服务平台。
- 自然资源行业标准优先采用自然资源标准化信息服务平台。
- 两个官方来源冲突时，保留冲突信息并提示用户，不静默覆盖。

### 7.9 用户上传与管理员审核

用户上传资料作为 V1 功能预留。

规则：

- 云端内测已经要求邀请码登录；上传功能继续复用现有账号体系。
- 用户上传资料默认只对上传用户可见。
- 用户可申请将资料提交管理员审核。
- 管理员审核通过后，资料才能进入受控服务可检索范围。
- 审核未通过的资料不能影响其他用户，也不能作为服务端共享答案依据。
- 用户提问触发的联网 OCR 或网页解析结果也应进入候选暂存区，不能自动进入后台正式知识库。

审核内容：

- 来源是否可靠。
- 标准号、名称、版本是否正确。
- 是否重复入库。
- OCR 或解析质量是否达标。
- 是否存在版权或使用边界问题。

候选暂存区要求：

- 保存触发问题、候选标准、官方来源 URL、采集时间和处理状态。
- 保存 OCR 文本、页码映射、置信度、OCR 工具和版本。
- 支持管理员按标准号、来源、置信度和创建时间筛选。
- 支持批量通过、拒绝、要求重新 OCR 或标记仅供内部使用。
- 审核通过后才写入正式知识库、全文索引、向量索引或图谱。

### 7.10 用户反馈

用户可以对答案反馈：

- 有用。
- 无用。
- 引用错误。
- 标准状态疑似错误。
- 没有回答到问题。
- 证据不足。

反馈用于后续改进检索、知识库和提示词。

反馈必须写入可查询的处理队列并关联 `request_id`、会话和原问题：

- `quote_too_long`、`answer_too_vague`、`format_issue` 等进入程序/问答优化通道，可由主开发 Agent 定位并修复。
- `wrong_standard`、`wrong_clause`、`missing_evidence` 等进入知识库审核通道，由管理员或 KB 专员确认后才能修改正式知识库。
- `other` 进入人工判断通道；满意反馈直接关闭。
- 管理员后台可以查看问题、用户、原因、处理通道和状态，并记录处理中、待 KB 审核、已解决或已忽略及处理说明。

### 7.11 微信登录与邀请好友奖励（待前置条件满足）

优先级：普通。

状态：规划完成，受前置条件阻塞，当前不得进入编码、联调或生产部署。

本阶段只增加微信开放平台网站应用登录和“邀请好友试用”奖励。支付宝登录、手机短信登录和手机号码验证均不属于本阶段范围，也不作为微信登录上线的阻塞项。

#### 启动前置条件

以下条件必须全部满足后，任务才能启动：

- geowiki 正式域名已经完成 ICP 备案，备案主体与实际运营主体、微信开放平台认证主体保持一致。
- 正式域名已经配置有效 HTTPS 证书，HTTP 自动跳转 HTTPS，生产环境会话 Cookie 可以启用 `Secure`。
- 微信开放平台账号已经完成所需认证，并已创建、审核通过“网站应用”。
- 已取得网站应用 `AppID`、`AppSecret` 和 `snsapi_login` 权限，授权回调域名与开放平台审核配置完全一致。
- 网站已经提供应用介绍、功能详情、用户协议、隐私政策、版权所有者、联系方式、备案信息和账号注销/解绑说明。
- 生产环境已经具备安全保存密钥、轮换密钥以及区分开发/生产回调地址的配置能力。

前置条件未全部满足时，只保留本 PRD 规划，不以临时 IP、HTTP 地址、未审核应用或模拟生产凭据提前上线。

#### 微信登录与账号绑定

- 使用微信开放平台“网站应用微信登录”的 OAuth2 authorization code 流程，不使用公众号网页授权替代。
- 登录发起端生成一次性随机 `state`，服务端以 Redis 或等价的短期存储保存并在回调时校验；校验成功后立即作废，防止 CSRF、重放和登录串号。
- `AppSecret` 只保存在服务端环境变量或密钥管理服务中，不写入前端、日志、数据库明文字段或 Git。
- 登录完成后，将微信身份映射到 geowiki 本地用户，再复用现有 Session、API Key、会话历史、每日配额和封禁状态。
- 新增独立第三方身份表，至少保存 `provider`、稳定身份标识、`openid`、可用时的 `unionid`、本地 `user_id`、绑定状态和时间；同一微信身份只能绑定一个本地用户。
- 不依据微信昵称、头像或相似显示名自动合并账号。已有用户必须先使用原账号登录，再主动绑定微信，避免产生不可逆的错误合并。
- 新微信用户允许没有密码，账号表需要支持可空密码和独立身份记录；管理员账号及原有用户继续保留邮箱密码登录作为故障兜底和账号恢复方式。
- 云端内测继续执行邀请码准入。微信扫码只替代身份验证步骤，不能绕过有效邀请码直接开放注册。
- 解绑微信前必须确认账号仍保留至少一种可用登录方式；封禁本地账号时，微信登录和邮箱登录必须同时失效。
- 登录、绑定、解绑、失败回调和身份冲突均写入安全审计记录，但不得记录授权码、访问令牌或 `AppSecret`。

#### 邀请好友试用

产品不使用“点击分享立即加次数”“转发朋友圈领取次数”或强制分享设计。分享按钮只能帮助用户发送链接，奖励必须依据服务端可核验的有效邀请结果发放。

有效邀请流程：

1. 已登录用户生成自己的邀请链接、邀请码或二维码。
2. 新用户首次通过该入口进入，服务端记录一次不可重复覆盖的邀请归因。
3. 新用户使用微信完成身份登录，并通过内测邀请码准入要求。
4. 新用户完成首次有效的地质领域问题，且请求形成正常回答；无关问题、系统错误、取消任务和明显滥用请求不计为有效邀请。
5. 奖励先进入待确认状态，经过默认 24 小时观察期和风险检查后再由服务端原子发放。

新用户试用配额：

- M14 上线后注册的新用户，包括通过邀请链接进入的用户，自注册成功时获得连续 30 天试用期。
- 试用期内每日提供 10 个免费配额单位，按 `Asia/Shanghai` 自然日重置，当日未使用额度不结转。
- 基本模式消费 1 个单位，深度模式消费 3 个单位；系统错误和符合退款条件的取消继续退回对应单位。
- 试用期满后，每日免费额度归零，账号和历史数据继续保留；用户仍可消费永久奖励余额或管理员单独追加的额度。
- M14 上线前已注册用户不自动追溯改为 30 天试用，继续保留既有额度策略，管理员可以按实际内测安排单独调整。

默认奖励规则：

- 每个有效邀请向邀请人发放 5 个永久配额单位，相当于 5 次基本模式，或者 1 次深度模式和 2 次基本模式。
- 每名邀请人最多获得 5 次有效邀请奖励，永久奖励累计上限为 25 个单位；达到上限后邀请入口显示“奖励已达上限”，不再接受新的奖励归因。
- 永久奖励余额不设置有效期，也不受每日 10 个免费单位上限约束；用户可以在同一天继续消费余额，但仍受 API 每分钟限流、深度任务并发和全局服务保护限制。
- 当日免费配额优先消费，免费配额不足时再消费永久奖励余额。深度模式必须原子取得 3 个可用单位，不能部分扣减后启动。
- 用户获得邀请资格前，需要满足可配置的账号存续时间和有效领域问答数量门槛；具体初始门槛在 M14 上线前由管理员确认。
- 活动设置全站月度新增用户上限、全站有效邀请上限和紧急暂停开关，防止邀请链在短时间内造成超出服务器承载能力的增长。
- 同一新用户只能归因一次、同一有效事件只能发奖一次，禁止自己邀请自己；奖励发放必须使用唯一幂等键和数据库事务。
- 仅打开分享面板、复制链接、扫码但未登录、重复注册或未完成有效领域问答均不发放奖励。
- 风险判断可以使用微信稳定身份、现有账号、请求 IP、时间模式和最小化的设备风险信号；不得为了反作弊过度采集个人信息，风险字段应限制保存期限并优先哈希存储。
- 可疑奖励可以进入人工审核、撤销或封禁流程，所有余额变化必须保留不可篡改的奖励账本和原因。

奖励数据不得复用仅对单个日期生效的 `daily_usage.bonus_count` 作为长期余额。至少需要独立的邀请代码、邀请归因、奖励账本和余额汇总结构，并保证网页问答与用户 API Key 调用消费同一账户余额。

#### 管理员入口

管理员后台需要提供：

- 微信登录功能状态、回调健康检查和不包含密钥的配置完整性提示。
- 用户已绑定的登录方式、绑定时间、状态和管理员强制解绑入口；强制解绑不得使账号失去全部登录方式。
- 新用户试用期天数、每日免费单位、邀请资格门槛、活动启停、观察期、单用户 5 次奖励上限和全站月度注册上限配置。
- 邀请次数、注册转化、首次有效问答转化、待确认/已发放/已撤销奖励和异常比例统计。
- 按用户查看邀请关系、奖励余额和完整账本，并支持填写原因后人工通过、撤销奖励或封禁账号。

#### 验收标准

- 微信登录、已有账号绑定、新用户邀请码准入、解绑和故障兜底均有自动化测试。
- 错误或过期 `state`、重复回调、回调地址不匹配和身份已被其他账号绑定时必须拒绝登录。
- 同一微信身份不会产生多个本地账号，原有邮箱用户、API Key、会话和配额数据不丢失。
- 前端分享回调不能直接发奖；并发重复事件、自邀和重复新用户均不能造成重复奖励。
- 新用户 30 天试用期、每日 10 个单位、自然日重置、不结转和到期归零均有边界时间测试。
- 每个有效邀请只发放 5 个永久单位，每名邀请人最多发放 5 次且累计不超过 25 个单位；达到上限后并发回调也不能继续发奖。
- 永久奖励无过期时间和每日消费上限；基本模式和深度模式可以按 1/3 单位依次消费当日免费配额与永久余额，失败退款和账本记录保持一致。
- 管理员所有配置、审核、撤销和封禁操作均有权限校验与审计记录。
- 隐私政策明确说明微信身份数据、邀请归因和风险数据的用途、保存范围及注销/解绑方式。

## 8. 答案策略

### 8.1 本地知识库优先

系统首先使用本地知识库中的已解析、已校验文档。

本地知识库命中条款级证据时，可以生成规范性回答。

### 8.2 联网只补充，不兜底编造

联网搜索只补充来源和元数据。若联网只找到元数据，不生成条款级结论。

### 8.3 OCR 结果需标记

如果正文来自 OCR 或视觉解析，应保留解析来源和置信度。

低置信度 OCR 文本不应作为强结论依据。

推荐 OCR 备选工具：

- PaddleOCR：优先作为中文标准、扫描 PDF、图片型 PDF 的 OCR 试验工具。
- PP-StructureV3：用于版面分析、章节结构识别、表格识别和复杂文档解析试验。
- TableRecognitionPipelineV2：用于表格较多的标准附录、指标表、分类表识别试验。

选型理由：

- 对中文识别支持较好。
- 支持文档结构解析和表格识别，适合标准、规范、规程类文件。
- 已在本地 Codex 专用虚拟环境中安装验证，可作为 MVP 后续 OCR 试验基础。

使用限制：

- OCR 结果必须保存页码、来源文件、模型版本和置信度。
- OCR 结果进入知识库前应抽样校验。
- 图片型标准全文的抓取、长期保存和对外展示需要先确认版权与使用边界。

### 8.4 第三方资料谨慎使用

第三方 PDF、网页、文库类页面只能作为候选来源。

除非经过校验，否则不能作为最终标准正文依据。

用户上传的第三方 PDF 默认仅对上传用户开放。经管理员审核通过后，才能进入受控服务可检索范围。

## 9. 数据来源策略

### 9.1 官方国标来源

国家标准可优先查询：

- `std.samr.gov.cn`
- `openstd.samr.gov.cn`

已验证问题：

- 部分国标在线预览为图片切片或视觉阅读器。
- 页面可提供元数据，但正文不一定能被文本模型直接读取。

### 9.2 官方行标来源

行业标准可优先查询：

- `std.samr.gov.cn/hb`
- `nrsis.org.cn`

已验证问题：

- `std.samr.gov.cn/hb` 可能只提供元数据。
- `nrsis.org.cn` 对自然资源领域行标可能提供全文阅读入口。
- `nrsis.org.cn` 阅读器可能按页返回图片型 PDF，需要 OCR。

### 9.3 本地知识库

本地知识库应保存：

- 原始文件。
- 解析文本。
- OCR 文本。
- 页码映射。
- 条款层级。
- 标准状态和版本。
- 来源 URL。
- 入库时间。
- 校验状态。

## 10. 知识库构建需求

知识库由独立服务或另一个 agent 负责构建。本产品后端只依赖知识库服务接口，不直接绑定其内部实现。

详细要求见 [KNOWLEDGE_BASE_REQUIREMENTS.md](/home/nalanmading/My-project/my-1st-agent/docs/KNOWLEDGE_BASE_REQUIREMENTS.md)。

PRD 层面对知识库的要求是：

- 知识库必须先建立稳定 schema，再做索引和检索。
- 知识库必须返回证据列表，不直接生成最终答案。
- 证据必须可追溯到文档、章节、条款或页码。
- MVP 阶段优先支持标准目录查询和 Elasticsearch 全文搜索。
- v1.0 使用 `text-embedding-v4` + USEARCH ANN 和 SQLite KG；ChromaDB/专用向量数据库、Neo4j 作为规模化替换选项。
- Embedding Provider 必须可配置，不绑定 `deepseek-v4-flash`。
- 知识库需要标记来源类型、正文可访问状态和质量状态。
- 当本地证据不足时，知识库应返回 `needs_web_supplement: true`，由问答后端或联网模块补齐官方元数据、全文入口或 OCR 任务。

### 10.1 接受的知识库架构

软件应明确声明支持的知识库架构。这样开源本地版、云端受控测试版和机构内部部署版都可以按同一套接口接入，agent 也能最大化利用标准号、条款、页码、来源、OCR 置信度和审核状态。

最低兼容架构：

- 结构化文档元数据。
- 条款或文本片段表。
- 页码或来源定位字段。
- 标准目录查询接口。
- 证据检索接口。
- 来源类型、正文可访问状态和质量状态字段。
- 领域术语库或同义词表，用于用户表达、标准术语和意图标签之间的映射。

推荐 MVP 架构：

- PostgreSQL 或等价关系型数据库：保存 schema、标准元数据、条款、页码、表格、上传和审核状态。
- Elasticsearch：提供标准号、术语、条文关键词等全文搜索。
- 对象存储或本地文件目录：保存用户上传文件、原始 PDF、OCR 中间结果，按授权策略决定是否长期保存。
- 后台任务队列：处理 OCR、解析、入库、质量检查。
- 统一知识库 API：至少提供 `/knowledge/search` 和 `/knowledge/standards`。
- `domain_lexicon`：保存同义词、规范术语、意图标签、适用场景和负面降权规则。

增强架构：

- USEARCH：v1.0 的私有 ANN 语义召回索引；后续可按规模替换为 ChromaDB、Milvus、pgvector 或等价服务。
- Embedding Provider：独立配置，不使用 `deepseek-v4-flash`。当前使用阿里云百炼/DashScope `text-embedding-v4`。
- SQLite KG：v1.0 已用于标准、矿种、条款、材料、时限和责任关系；Neo4j 是后续规模化选项。
- 按需混合检索：全文搜索、结构化字段和知识图谱优先；证据不足时再补充向量召回并统一排序。
- Reranker：用于证据重排序和可回答性判断，优先判断“证据是否回答了用户意图”，而不是只看语义相似度。

如果用户已有自己的知识库，只要能通过适配器返回本系统要求的证据格式，也可以接入；但缺少 schema、条款定位、页码映射和质量状态时，agent 只能降级回答。

## 11. 非功能需求

### 11.1 准确性

专业答案必须以证据为核心。系统宁可回答“不足以判断”，也不能编造条款。

### 11.2 可追溯性

每个结论应尽量追溯到文档、页码、章节或条款。

### 11.3 响应速度

v1.0 目标：

- 已验证的结构化常规问题 2 秒内返回。
- 需要模型规划和证据审查的复杂比较问题目标 30 秒内返回。
- 需要联网补充的问题 30 秒内返回初步结果。
- OCR 不要求同步完成，可进入后台处理。

### 11.4 安全与合规

- 不提交 `.env` 或 API Key。
- 不公开知识库本体、标准全文库、OCR 全文、索引、数据库或 `/knowledge/*` 后台检索服务。
- 尊重官方平台的使用提示和版权声明。
- 对外展示内容以必要、限长、可核验的引用片段、摘要和来源说明为主。
- 公开社区版不得包含真实标准全文、OCR 后全文、预构建真实标准知识库或向量库。
- 机构内部部署版可按资料授权范围配置长文本返回、全文索引和内部导出。
- 云端受控测试版只开放问答 API 输出，不提供知识库下载、镜像、全文下载、批量导出、大段原文展示或直接知识库检索入口。
- 公共 API 必须具备 API Key、每日配额、限流、调用日志和用量统计。
- 限流优先采用 Redis 作为共享计数后端，支持多进程和未来多节点部署；Redis 不可用时应降级为本地内存限流，避免开发和单机内测被外部依赖阻断。
- 无关问题必须在低成本阶段拒答，避免被当作通用大模型接口消耗 token。
- 恶意调用、异常高频调用、提示词注入和批量探测请求应进入风控统计，必要时限流或封禁 API Key。

### 11.5 成本控制

系统应把成本高的操作放在后置流程。

要求：

- 先做领域相关性判断，再进入检索和模型推理。
- 先查正式知识库，再决定是否联网补充。
- OCR、多模态理解、长上下文推理默认异步执行。
- 知识库缺口任务可排队慢处理，不要求并发。
- 后台补库优先处理高频、明确、领域相关的问题。
- API 返回应包含配额消费和剩余次数，便于用量控制、审计和防滥用。

### 11.6 可维护性

- 检索服务、联网补充服务、OCR 服务和 LLM 生成服务应解耦。
- 来源分类、答案策略和提示词应可配置。
- 后续可以替换模型供应商。

## 12. 部署与开放形态

产品支持三种部署形态，详见 [LICENSING_AND_REPOSITORIES.md](/home/nalanmading/My-project/my-1st-agent/docs/LICENSING_AND_REPOSITORIES.md)。

### 12.1 开源本地版

开源本地版面向个人、研究者和小团队内部试用。

边界：

- 采用 AGPL-3.0。
- 用户自行导入合法持有的资料。
- 仓库不内置真实标准全文。
- 可支持本地长文本返回，由用户自行承担资料使用边界。

### 12.2 机构内部部署版

机构内部部署版面向依法持有资料的单位内部使用，当前不提供收费交付。

可提供：

- 企业权限和组织级知识库。
- 管理员审核后台。
- 审计日志。
- 企业私有模型和内网模型适配。
- 高可用部署。
- 定制知识图谱。
- 技术支持、培训和维护服务。

### 12.3 云端受控测试版

云端受控测试版采用 API-first 形态。前端主要作为内测入口、API 调试台和管理员后台，不设置充值、订单、套餐或其他盈利功能。

知识库作为产品护城河，在任何部署形态下都不对外公开。云端受控测试版只对外提供受控问答 API、任务状态和必要来源定位。

- 不提供官方标准全文公共库或任何形式的知识库本体公开访问。
- 不提供大段原文展示。
- 不提供全文下载或批量导出。
- 用户上传资料默认私有。
- 服务可检索资料必须经过审核或授权。

受控 API 能力：

- 邀请码注册、账号登录和浏览器会话。
- API Key 管理。
- 网页问答与 API 调用共用账号每日次数配额。
- 调用限流，优先 Redis，支持本地内存兜底。
- 用量统计。
- 调用日志和审计。
- 结构化错误码。
- OpenAPI 文档和 SDK 示例。
- 任务状态查询，例如知识库缺口任务、OCR 任务和候选入库任务。

每日配额规则：

- 新用户默认每天拥有 10 个问答配额单位，日期按 `Asia/Shanghai` 计算。
- 网页会话和同一账号创建的全部 API Key 共享配额。
- 基本模式每次原子预留 1 个单位；`answered`、`insufficient_evidence` 和 `queued_for_enrichment` 结算为消费 1 个单位。
- 深度模式创建任务时原子预留 3 个单位；可用配额不足 3 个单位时不得创建任务。
- 同一问题已经在基本模式消费 1 个单位后，由该答案一键转为深度模式时只追加预留 2 个单位，使该问题总消耗为 3 个单位；后端必须校验原请求归属、问题和会话，不能由客户端自行声明优惠。
- 深度任务的 `completed`、`partial` 和 `insufficient_evidence` 均消费预留单位；知识库覆盖不足属于研究结果，不属于系统异常。
- 深度任务发生 `failed` 系统错误，或仍处于 `queued` 时由用户取消，应全额退回本次深度任务预留单位。
- 同一用户同一时间最多有一个 `queued`、`planning`、`retrieving` 或 `analyzing` 状态的深度任务。
- `out_of_scope` 不消费次数，并且不得调用知识库、联网、OCR 或长模型推理；仍受账号、IP/API Key 的分钟级限流和审计约束，防止无成本滥用。
- `system_error` 退回预留次数。
- 管理员可以修改用户长期日上限，也可以给指定日期增加额外次数；所有调整记录操作人、原因和时间。
- 配额必须由后端以“单位数”事务原子预留和结算，兼容历史单次记录；前端只展示后端返回的消费单位和剩余单位。

## 13. 成功指标

MVP 验收指标：

- 用户能提交问题并获得结构化答案。
- 至少展示 1 条来源或明确说明无来源。
- 条款级回答必须包含引用。
- 无证据时不生成确定性结论。
- 用户反馈可被记录、分类并在管理员队列中闭环处理。
- 标准状态冲突可被提示。
- 无关问题能被拒答，且不进入补库任务。
- 知识库缺口问题能生成异步补库任务。

运营观察指标：

- 问题成功回答率。
- 有条款级引用的答案比例。
- 用户反馈“有用”的比例。
- 引用错误反馈比例。
- 证据不足问题 Top N。
- 本地知识库缺失标准 Top N。
- 无关问题拦截数量和比例。
- 知识库缺口任务完成率。
- OCR 候选审核通过率。
- API 调用量、平均成本和平均响应时间。
- 按意图统计前 5 条候选的同文档比例和语义重复率。
- 正确证据已召回但未进入最终引用的比例。
- 查询改写前后直接证据命中率，以及第二轮 `refined_query` 的证据补全率。
- Multi-Query 每增加一次子查询所带来的新增直接证据数、重复候选数、模型成本和延迟。

### 14.1 领域词典治理（v2.1.0 已实施）

领域词典不能依靠自动挖掘结果直接上线。系统可以从用户反馈、失败问题、知识库 Schema 或管理员录入中生成候选，但以下内容必须由管理员人工决策：

- 用户表达应映射到哪个规范术语、意图和业务领域。
- 该表达是否足够明确，可以参与领域门控；通用办事表达必须配置必要上下文和禁止上下文。
- 该词条只用于检索扩展，还是可以触发确定性意图；两类职责使用独立开关。
- 应增加哪些正向扩展、负面降权词和证据必须满足的表达。
- 正向示例是否稳定命中，容易误判的反例是否被正确排除。
- 风险等级、优先级、批准、驳回、停用和恢复决定。

管理员操作闭环：

1. 从“领域词典”页面新增候选，或在回答反馈中选择“转词典候选”。
2. 填写规范映射、运行开关、上下文、检索/证据约束、正向示例和反例，并将状态设为“待审核”。
3. 使用“预览命中”对比当前规则和候选生效后的领域门控、意图、检索扩展及证据约束；系统同时自动验证全部正例与反例。
4. 只有最新候选指纹的正反例预览全部通过，且填写审核记录后，“批准并生效”才可执行。
5. 批准后原子发布运行时词典；正式词条的修改必须再次生成候选。停用或恢复必须填写原因，并立即写入审计记录。

后端必须重复执行上述约束，不能只依赖浏览器按钮状态。任何影响匹配行为的修改都会使旧预览失效；绕过前端直接调用管理员 API 也不能批准未预览、正反例失败或仍为草稿的候选。运行时词典、候选、审核记录和用户反馈属于后台运营数据，不公开、不提交 Git。

后续管理页面继续增加：

- 无关问题拦截记录。
- 知识库缺口任务列表。
- OCR 候选暂存区。
- 管理员审核入口。

域名备案、HTTPS 和微信开放平台网站应用审核全部完成后，按 7.11 和 M14 增加：

- 登录页微信扫码入口、扫码状态、失败重试和邮箱登录兜底。
- 账号安全页的微信绑定、解绑和登录方式状态。

页面风格偏专业工具，不做营销落地页。重点是问答效率、证据核验、调用接入和运营可观察性。桌面和移动端均需避免横向溢出、文本遮挡和控件错位。

### M13: Prompt 校准层评估

优先级：低。

- 建立 30 条版本化离线校准样本与 `scripts/evaluate_prompt_registry.py` 覆盖校验；v3.0.0 受控真实模型评估中，基线与校准变体的分类准确率均为 100%（30/30），基线 P95 为 4047.49 ms，校准变体 P95 为 4653.50 ms。校准变体没有带来准确率提升且增加延迟，因此生产环境保持关闭；不在生产主链路中用 Prompt 掩盖知识库或召回缺口。

- 分别建立查询规划、证据审查和答案表达校准集。

#### Schema 使用原则

Schema 是 geowiki 的必要基础设施，但不是答案来源，也不是唯一召回通道。系统至少维护知识库 Schema（文档、适用范围、专业主题、条款层级、来源与质量）、问题 Schema（意图、对象、矿种、阶段、事项、条件和必需证据槽位）及证据 Schema（结论必须覆盖的条款、条件、来源和不确定性）。

Schema 的职责是缩小方向、展开结构、校验证据和记录诊断；FTS、向量检索和知识图谱负责多路召回；大模型负责自然语言理解和受证据约束的归纳。用户明确限定的标准号、条款号、数值、阶段和事项属于硬约束；适用范围、主题关系和候选标准属于软路由提示。软 Schema 范围内证据不足时必须保留全库混合检索回退，禁止因 Schema 缺项排除已入库的正确正文。

当前问题暴露的是 Schema 维度不足和证据门禁过弱，而非“Schema 导致检索效果差”。后续优先补充标准适用范围关系、条款父子层级、条件矩阵和章节级证据组；不得把单题答案、固定标准号或大规模同义词硬编码进 Schema。

- 对该类别启用证据审查或等效的确定性关系校验，禁止仅凭一条含“应/必要时”的条款作出替代性结论；检索轨迹记录阶段要求证据、层级关系证据、是否存在用户未明示假设及最终关系判定。

v3.0.1 实施结果：新增受保护内部检索意图 `technical_requirement_sufficiency`，识别满足、替代、覆盖和“是否还必须做”等技术关系表达；对“实验室流程实验”进行无语义损失的标准术语归一。该策略不绑定矿种或标准号，默认检索阶段要求与研究层级两类证据。标题块不再能够单独通过证据门禁；回答前必须同时取得阶段条款和直接的“以另一试验/测试为基础”的层级条款。第一语义阶段、证据审查和最终回答 Prompt 均明确禁止补入用户未陈述的“未完成较低级工作”事实。真实私有库验证已同时命中 DZ/T 0205-2020 4.3.3 与 DZ/T 0340-2020 5.2.2；全量自动化回归通过。

- **等级满足判断**：用户问“半工业试验能否满足详查阶段要求”“实验室扩大连续试验能否覆盖实验室流程试验”等，默认用户所述试验已经是合格的对应等级试验。系统只比较同一技术体系内的标准研究等级；实际等级高于或等于要求等级时，直接判定满足，不再附加“还需证明该高等级试验包含低等级试验”的条件。
- **试验符合性核验**：用户问“这项试验能否算半工业试验”“样品重量、连续时长、流程、采样是否符合半工业试验要求”等，才检索和核验试验规模、设备、样品代表性、运行时长、平行试验、工艺参数、记录及环境要求。该模式的结论不能反向影响等级满足判断中对用户事实的默认接受。

选冶试验的等级 Schema 应维护带标准条款来源的有向层级：`可选性试验 -> 实验室流程试验 -> 实验室扩大连续试验 -> 半工业试验 -> 工业试验`。等级满足判断沿同一链条进行传递闭包；工艺矿物学研究、物化性能测试等属于不同证据轴，不应因名称看似“更高”而自动互相覆盖。后续回答只需说明适用的阶段、要求的最低等级、用户已完成等级及等级比较结论；除非用户主动询问符合性，否则不展开样品质量、时长和流程等核验要求。

v3.0.2 实施结果：

- 新增受条款来源约束的选冶试验等级 Schema，并以等级序号执行同一体系内的覆盖判断；不再由最终回答模型自行推断“更高等级是否覆盖较低等级”。
- `technical_requirement_sufficiency` 对用户已明确的试验等级采用事实前提，不附加样品、设备、时长、采样或记录的符合性条件；“半工业试验能否满足实验室流程试验要求”直接按等级关系给出结论。
- 新增受保护的 `technical_test_conformity_verification` 检索意图。只有“能否算某等级试验”或明确询问样品、设备、时长、处理量、采样、记录等问题时，才进入具体符合性核验。
- 修复“工业试验”被“半工业试验”包含时的子串误识别；等级提取按最长规范术语消除重叠。
- 用户明确目标等级时，不再为补充阶段背景而引用其他矿种的阶段条款；等级关系依据只引用对应等级 Schema 的来源条款。
- 领域词典的 `background_context` 只扩展术语，不再把其他业务意图的证据约束注入当前查询，避免无关检索污染。
- 已完成等级覆盖、低等级不足、符合性分流、术语重叠和既有检索链路的自动化回归。

## 16. 已确认产品决策

- 产品显示名为 `geowiki`，中文副标题为“一款专注地质领域的百科全搜”。现有 Python 包名、部署路径和 systemd 单元暂时保留，避免无关迁移风险。
- 云端内测采用邀请码、邮箱验证码注册和登录，不开放匿名问答；网页和用户 API Key 共用账号每日次数。
- 微信登录与邀请好友奖励列为普通优先级 M14，但只有在正式域名 ICP 备案、HTTPS、微信开放平台认证和网站应用审核全部完成后才能启动；前置条件满足前继续使用现有邮箱登录体系。
- 微信登录上线后仍保留邮箱密码作为管理员登录、账号恢复和平台故障兜底；内测期微信新用户仍需邀请码，扫码登录不能成为公开注册入口。
- “分享得次数”采用服务端核验的“邀请好友试用”机制，不按点击、复制链接或前端分享回调发奖；每个有效邀请奖励 5 个永久配额单位，每名邀请人最多获得 5 次奖励、累计上限 25 个单位。
- M14 上线后注册的新用户获得每日 10 个免费配额单位，持续 30 天，按北京时间自然日重置且不结转；永久邀请奖励无有效期、不受每日免费额度上限影响，但继续受接口限流、任务并发和全局服务保护约束。
- 支付宝登录、手机短信登录和手机号码验证暂不考虑，不属于 M14 范围。
- 登录用户的会话历史保存在独立应用数据库中，用户可以删除会话；知识库数据库不保存用户账号和配额数据。
- M14 启动前继续沿用现有用户每日 10 个配额单位策略；M14 上线后按新用户 30 天试用和永久邀请奖励规则执行，M14 上线前已注册用户不自动追溯调整。基本模式消费 1 个单位，深度模式消费 3 个单位，基本答案一键升级只追加 2 个单位。系统错误和排队阶段取消退回对应预留单位。系统不提供充值、订单或支付功能。
- 基本模式用于明确条款、术语、数值、材料和来源的快速查证；深度模式用于跨文档完整性审查、差异对比和复杂条件分析，两者均不得用模型常识替代知识库证据。
- 问题歧义判断以大模型为主，业务 Schema 和受保护规则负责最终校验，领域词典只承担核心术语、业务等价关系、反例和模型不可用时的降级，不再以穷举词库替代自然语言理解。
- 意图确认不消耗问答次数；确认后的完整问题才进入本地知识库检索。网页与 API 使用同一结构化确认协议。
- 引用展示优先包含标准名称、标准号、具体条款和条款原文；页码作为可选核验字段保留。
- v1.0 知识库已完成 schema、标准目录、FTS5、SQLite KG、`text-embedding-v4` 与 USEARCH ANN；索引更新必须与私有 SQLite KB 同步发布。
- 生成模型使用 `deepseek-v4-flash`；embedding 使用独立 provider，不使用 `deepseek-v4-flash`。默认配置为 `EMBEDDING_PROVIDER=aliyun`、`EMBEDDING_MODEL=text-embedding-v4`、`EMBEDDING_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`。
- 在线 embedding 只在文档入库、补库或批量重建时产生模型服务商调用成本；普通问答只需对用户问题做一次 embedding。批量重建真实知识库向量前需要人工确认成本和批量范围。
- 普通问答并非固定调用 embedding：候选标准已锁定且全文/图谱证据足够时应直接跳过；跨标准比较或证据不足时才生成查询向量。
- 官方图片式预览允许 OCR 入库，但来源限定为已确认官方平台，优先是国家标准全文公开系统、全国标准信息公共服务平台和自然资源标准化信息服务平台。
- OCR 文本进入知识库后需要长期保存 OCR 文本、页码映射、置信度、OCR 工具和版本；原始文件是否长期保存按授权和存储策略决定。
- 用户上传第三方 PDF 默认只对上传用户可见，经管理员审核通过后才能进入受控服务可检索范围。
- 标准状态冲突时，国标优先采用国家标准官方平台，自然资源行业标准优先采用自然资源标准化信息服务平台，同时保留冲突提示。
- 社区版继续采用 AGPL-3.0。此前规划的商业许可证路线当前暂停，不属于云端受控测试版功能。
- 私有仓库可继续用于保存不公开的内部部署代码和配置，但不得包含知识库明文或密钥。
