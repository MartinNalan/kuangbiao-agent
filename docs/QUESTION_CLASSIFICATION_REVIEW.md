# geowiki 问题分类与检索策略确认表

## 1. 本表的作用

本表定义系统应当如何识别一类问题，以及识别成功后如何调用知识库。它不是具体问题的标准答案表，也不会用于给某几个问题硬编码答案。

用户确认本表后，系统将据此建立统一的 `QueryClassification` 和检索策略注册表。后台已有反馈问题只作为测试样本，用于检查分类和检索策略能否处理没有提前穷举的新问法。

## 2. 填写方法

本表已由主开发 Agent 按当前产品目标完成第一版填写。请直接审阅并修改任意单元格，重点检查以下内容：

- “常见表述”：补充行业人员真实会使用的口语、简称和不同问法。
- “何时必须追问”：判断缺少什么信息时确实不能直接检索或会改变结论。
- “建议检索策略”：确认系统应该优先查哪类文件、采用什么检索顺序。
- “初步决策”：检查我对默认行为、分类边界和产品取舍的判断是否符合实际业务。

表中其他内容是系统建议，也可以直接修改。无需为每个类别穷举所有可能问题，重点是明确业务目标、决定结论的字段和稳定的检索方法。

## 4. 所有分类共同遵守的规则

- 大模型负责理解用户真正要完成的事情、纠正错别字和口语、识别分类及抽取槽位，但不在检索前生成专业结论。
- 标准号、文号、矿种、数值、用户明确限定的对象和否定纠正语义由后端校验，不允许模型删除或擅自改变。
- 领域词典在分类完成后使用，只补充当前类别下的同义词和检索表达，不能决定主要分类，也不能跨类别注入无关证据词。
- 基本模式和深度模式共享同一个 `QueryClassification`。深度模式只能扩大候选范围和审查深度，不能重新判断用户问的是什么。
- 一个问题可以有一个主要分类和多个次要分类。例如“采矿权延续需要哪些材料，办理流程是什么”以 `service_materials` 为主要分类，同时包含 `service_workflow`。
- 具体问题和标准答案只用于回归验证。新增问题即使表达方式从未出现过，也应通过类别描述、语义理解和槽位抽取得到正确策略。

用户完成本表后，开发工作是把确认后的分类体系实现为统一数据结构和策略注册表，并使用后台真实反馈检查泛化效果，而不是为反馈题逐题增加答案模板。
